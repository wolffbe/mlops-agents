\chapter{Background}

This chapter presents the foundational concepts that underpin this study. As the thesis examines \gls{ML} platforms, it begins with an overview of \gls{AI} and its subfields, then moves from \gls{ML}, where the core learning paradigms of modern model training are introduced, to \gls{ML} engineering, emphasizing practical methods and tools for building models, and further to \gls{MLOps}, with its focus on automation and scalable operation of the \gls{ML} lifecycle. Finally, it introduces \gls{ML} platforms, the infrastructure that brings together and supports all of these practices.

The chapter then explores \glspl{LLM} in the context of \gls{SWE}, beginning with \gls{DL}, the subset of \gls{ML} that enables the neural network architectures behind modern generative models, and \gls{GenAI}, the theoretical foundation for models that generate new content rather than merely analyzing existing data. \Glspl{LLM} are examined as the specific class of generative models driving code generation, followed by code generation models as specialized variants, and vibe coding as an emerging development practice that leverages these models for \gls{SWE} tasks. \Gls{AutoML} is then discussed as the application domain in which \glspl{LLM} are increasingly employed to automate the construction of \gls{ML} pipelines.

As this research focuses on coding agents that use \glspl{LLM} to operate \gls{ML} platforms, it is essential to situate them within the broader field of intelligent agents. For that, intelligent agents and the theoretical principles of autonomy, reactivity, and proactiveness that define agent behavior are examined next. \Gls{DAI} and multi-agent systems are discussed to establish coordination and planning principles for environments where multiple agents or agent components collaborate. Software agents are then introduced as the bridge between theory and practical implementation, leading to coding agents, the class of \gls{LLM}-based \gls{AI} agents central to this thesis. Context engineering is also covered, as the effectiveness of coding agents depends heavily on how information is structured and presented within the context window, directly influencing interactions with external tools and platforms.

Building on these foundations, the chapter examines how coding agents interact with external systems. It reviews the evolution of tool use from early self-supervised approaches through function calling to the \gls{MCP}, a protocol that standardizes how agents discover and invoke external capabilities, and the \gls{CLI}, an interface that enables agents to operate directly within the local operating system. The \gls{CLI} is examined from its early development in interactive computing systems and early command interpreters, through the Bourne Shell, to modern research on natural language to \gls{BASH} translation and the design properties that explain the convergence of agentic coding tools on terminal-based interfaces. Finally, skills are reviewed as reusable behavioral abstractions that guide agent planning and execution, culminating in agentic coding manifests that provide agents with persistent project context and operational rules.

The subsequent section reviews related work across three areas, with the aim of identifying gaps in current research that motivate the \glspl{RQ}. The first examines \gls{ML}, covering both approaches to specializing software agents for \gls{ML} workflows and the broader landscape of \gls{ML} platforms. The second addresses existing benchmarks for coding agents, along with the \gls{MCP}, the \gls{CLI}, and \gls{ML} agents. The third focuses on the design and optimization of tools available to agents. The theoretical part concludes with a synthesis of the background and related work, from which the \glspl{RQ} guiding this thesis are derived.

\section{Artificial Intelligence}
\glsresetall

\Gls{AI} is a broad term describing “algorithms capable of performing tasks that typically require human intelligence, such as understanding natural language, recognizing patterns, making decisions, and learning from experience”~\cite{banh_generative_2023}. Early \gls{AI} systems, including expert systems and knowledge bases, were rule-based and designed to support users and businesses in decision-making processes. \Gls{ML}, a subfield of \gls{AI}, focuses on algorithms that learn to solve tasks from data without explicit programming. \Gls{DL}, in turn, is a subset of \gls{ML} that leverages neural networks to learn data representations and detect correlations and patterns in large datasets.

While \gls{ML} and \gls{DL} primarily focus on analyzing and learning from data, \glspl{DGM} aim to capture the underlying distributions of a dataset to generate outputs that closely resemble real-world data~\cite{banh_generative_2023}. When built on \gls{DL} architectures, these models are considered \gls{GenAI}. Generative modeling allows these models not only to replicate data patterns but also to understand the structure of the information and produce coherent, structured outputs. This capability is particularly effective in domains such as \gls{NLP}, where models are trained to comprehend and generate natural language. Among the most prominent examples are \glspl{LLM}, such as \gls{GPT}, which apply generative modeling to language and are increasingly used for software code generation. This has led to a growing ecosystem of code generation models, either adapted from general-purpose \glspl{LLM} or purpose-built for \gls{SWE} tasks. Development practices that rely primarily on these models for code generation are commonly referred to as vibe coding.

\subsection{Machine Learning}

\Gls{ML} is defined as ``a set of methods that can automatically detect patterns in data, and then use the uncovered patterns to predict future data, or to perform other kinds of decision-making under uncertainty''~\cite{murphy_machine_2013}. \Gls{ML} methods are commonly divided into supervised learning, unsupervised learning, semi-supervised learning, and \gls{RL}.

Supervised learning involves learning a function that maps an input to an output based on labeled examples. A training dataset consisting of input–output pairs is used to infer this mapping~\cite{sarker_machine_2021}. In that dataset, each input is represented as a $d$-dimensional vector of features. Features correspond to a measurable property of the input~\cite{murphy_machine_2013}. For example, in a weather prediction task, a single day could be represented as two features, the temperature and the day of the year, allowing the model to learn seasonal trends. Supervised learning is usually applied when target outcomes are known. Common tasks include classification, which assigns inputs to discrete categories such as ``rainy'' or ``not rainy'', and regression, which predicts continuous values such as the temperature in degrees.

Unsupervised learning, in contrast, analyzes data without labeled outputs or explicit human guidance~\cite{sarker_machine_2021}. Its objective is to identify patterns, structures, or relationships within the data through techniques such as clustering, dimensionality reduction, and anomaly detection. For example, clustering can group customers with similar purchasing habits, while dimensionality reduction can identify key factors in climate data.

Semi-supervised learning combines labeled and unlabeled data, which is useful when labeled data is limited or costly to obtain~\cite{sarker_machine_2021}. By using the structure of unlabeled data alongside available labels, the aim is to improve predictive performance. Applications include machine translation, fraud detection, and image classification where only a subset of images is labeled.

\Gls{RL} takes a different approach, involving an agent that learns to make decisions through interaction with an environment~\cite{arulkumaran_deep_2017}. The agent receives rewards or penalties based on its actions and seeks to learn a policy that maximizes cumulative reward over time. This approach is well suited for sequential decision-making problems such as robotics, autonomous driving, and game playing, but is less suited for simple prediction tasks.

While these learning paradigms provide the theoretical foundation for \gls{ML}, applying them effectively in practice requires a set of engineering principles, a structured lifecycle, as well as tools and organizational practices.

\subsubsection{Machine Learning Engineering}

Several practical principles are essential for the effective application of \gls{ML}~\cite{domingos_few_2012}. Learning is determined by the combination of representation, evaluation, and optimization, with generalization being the primary measure of success. While large datasets contribute to improved performance, data alone is not sufficient, and feature engineering often has a greater impact than algorithm choice. A central challenge in \gls{ML} is overfitting, which occurs when a model learns the training data too closely and fails to generalize to new, unseen data~\cite{chen_xgboost_2016}. Overfitting can manifest as bias, when a model captures incorrect patterns, or variance, when it captures random noise. Simple algorithms can outperform more complex ones under computational constraints, and ensembles of models generally achieve better results than individual models. For example, bagging combines multiple decision trees trained on different random subsets of the data to reduce variance, while stacking combines predictions from multiple models to improve overall accuracy.

Understanding these principles is critical not only for building models but also for ensuring that their behavior can be trusted and communicated~\cite{murdoch_definitions_2019}. Interpretability refers to the extent to which a human can comprehend how and why a model produces its outputs. It can be described along three dimensions. Predictive accuracy measures how well the model approximates the true underlying process. Descriptive accuracy evaluates how faithfully an explanation reflects what the model has actually learned. Relevance considers whether the explanation provides meaningful insights for a specific audience.

The \gls{AI} lifecycle, also referred to as the \gls{ML} lifecycle, is usually organized in three phases~\cite{de_silva_artificial_2022}. The first is the design phase, typically managed by a data scientist, and encompasses identifying and formulating the problem, reviewing data and \gls{AI} ethics, reviewing technical literature on \gls{AI} algorithms, applications, and pre-trained models, as well as data preparation, data exploration, and external data acquisition. The second is the development phase, carried out by \gls{AI}/\gls{ML} scientists, which involves data pre-processing, building an initial model, augmenting data, developing a benchmark, building multiple models, evaluating primary metrics, and explaining or interpreting the model. The third is the deployment phase, managed by the \gls{AI}/\gls{ML} engineer, and consists of evaluating secondary metrics, deploying the model and assessing risks, reviewing the deployment, operationalizing the model, and monitoring and evaluating the model's performance.

The most popular programming language in \gls{ML} is Python~\cite{raschka_machine_2020}. Python is a high-level interpreted programming language, meaning it abstracts away most low-level hardware details and executes code line by line rather than compiling it into machine code~\cite{van_rossum_python_2003}. Python supports an open-source ecosystem of frameworks for \gls{ML} engineering, including data preparation, model training, and visualization~\cite{raschka_machine_2020}. Key tools include NumPy for array programming and mathematical abstractions~\cite{harris_array_2020}, Pandas for data analysis and manipulation, scikit-learn and TensorFlow as \gls{ML} libraries~\cite{abadi_tensorflow_2016}, NetworkX for graph analytics, and Matplotlib for visualization~\cite{barrett_matplotlib_2005}.

In addition to programming, successful \gls{ML} engineering requires attention to broader technical and organizational concerns~\cite{serban_adoption_2020}. These span data management, training, coding, deployment, team collaboration, and governance, and include practices such as performing sanity checks on external data sources, applying version control to data, models, and training scripts, automating deployment processes, monitoring model behavior continuously, and ensuring fairness, accountability, and data privacy throughout the \gls{ML} lifecycle. Ensuring that these practices are consistently applied across the lifecycle of production-ready \gls{ML} products is the focus of \gls{MLOps}.

\subsubsection{Machine Learning Operations}

In around 2008, DevOps, a portmanteau of development and operations, emerged as a paradigm aimed at reducing the gap between development and operations teams by promoting collaboration, communication, and knowledge sharing~\cite{kreuzberger_machine_2023}. Through practices such as \gls{CI/CD}, which support continuous testing, quality assurance, monitoring, logging, and feedback loops, the goal of DevOps is to enable fast, frequent, and reliable software releases. Within this paradigm, software engineers are responsible for developing and operating systems, rather than either of the two. In around 2020, a similar paradigm emerged within the \gls{ML} community, focusing on automating and operationalizing the \gls{ML} lifecycle, which led to the inception of \gls{MLOps}.

\Gls{MLOps} is defined as the process of producing “\gls{ML} applications based on code, data, and models in small and safe increments that can be reproduced and reliably released at any time, enabling short adaptation cycles”~\cite{makinen_who_2021}. Systems involved in \gls{MLOps} can be categorized into three types: \gls{ML} platforms, \gls{ML} applications, and \gls{ML} automation frameworks. \Gls{ML} platforms provide the architecture to operationalize the \gls{ML} lifecycle through automated \gls{ML} pipelines. \Gls{ML} applications are custom pipelines designed to serve a domain- or industry-specific purpose, such as drug discovery. \Gls{ML} automation frameworks are specialized tools that enable the automation of individual stages within these pipelines~\cite{testi_mlops_2022}.

\Gls{ML} automation frameworks support different parts of the \gls{ML} lifecycle, including data pre-processing, model training, model management, model deployment, and operations and monitoring. Popular frameworks include MLflow, Airflow, Great Expectations, GitHub Actions, Kibana, and Grafana~\cite{ruf_demystifying_2021}. MLflow is an open-source platform for experiment tracking, model registry, and lifecycle management~\cite{zaharia_accelerating_2018}. Airflow is an open-source task orchestration platform that schedules and manages complex workflows~\cite{the_apache_software_foundation_airflow_2026}. Great Expectations is a framework for profiling, validating, and documenting data~\cite{gong_great_2026}. GitHub Actions is a component of GitHub, a web-based platform for collaborative software development using Git~\cite{github_inc_build_2026}, that automates software development workflows~\cite{github_inc_github_2026}. Git is a distributed version control system that tracks changes to source code, enabling multiple developers to work on the same codebase simultaneously while maintaining a complete history of modifications~\cite{the_linux_foundation_git_2026}. Kibana and Grafana are tools for visualizing and analyzing machine logs~\cite{elasticsearch_bv_kibana_2026, grafana_labs_grafana_2026}.

As \gls{MLOps} defines the practices and automation frameworks for operationalizing \gls{ML}, the effective implementation of these practices depends on the \gls{ML} platforms that support them.

\subsubsection{Machine Learning Platforms}

Due to the increasing importance of managing large quantities of data in \gls{ML}, an alternative perspective on the lifecycle emerged, positioning data-centric \gls{ML} as an independent phase preceding model-centric \gls{ML} and \gls{MLOps}~\cite{singh_systematic_2023}. Data-centric \gls{ML} consists of collecting, analyzing, and understanding data, including data quality checks, data augmentation, labeling, and data versioning. This leads to model-centric \gls{ML}, which involves initial model training, hyperparameter optimization, and model retraining. Hyperparameter optimization refers to the process of systematically tuning parameters that govern the learning process in order to improve model performance~\cite{bergstra_random_2012}. Examples include the learning rate, which controls the step size the model takes when updating weights during training~\cite{polyak_methods_1964}, and regularization strength, which limits the complexity of the model to prevent overfitting~\cite{hoerl_ridge_2000}. Finally, \gls{MLOps} encompasses model deployment, model interpretability, and model monitoring. A core part of monitoring is measuring skewness and detecting drift, which closes the loop back to data-centric \gls{ML} by triggering the reevaluation of new or changed data to ensure continued model performance.

Skewness is important in machine learning because it reveals asymmetry in feature distributions that can violate model assumptions, amplify outliers, and reduce predictive performance if not addressed~\cite{hastie_elements_2009}. Drift refers to gradual changes in data distributions over time~\cite{jim_dowling_building_2025}. Four types of drift are commonly recognized. Data ingestion drift occurs when the distribution of newly ingested features or labels significantly differs from existing data. Feature drift refers to changes in the distribution of a recent batch of feature data compared to the model's training dataset. Concept drift occurs when the relationship between input features and the target label changes over time, causing the model to lose accuracy. Prediction drift refers to a change in the distribution of a recent range of model predictions compared to predictions on the training dataset.

Given the central role of data throughout the \gls{ML} lifecycle, \gls{ML} platforms are fundamentally concerned with data management, aiming to provide highly available infrastructure for managing feature data with unified \gls{API} support~\cite{de_la_rua_martinez_hopsworks_2024}. An \gls{API} is a set of rules and protocols that allows different software systems to communicate and interact with each other in a standardized way~\cite{jim_dowling_building_2025}. One such platform is the Hopsworks Feature Store shown in Figure~\ref{fig:hopsworks}, which supports columnar, row-oriented, and similarity search workloads, as well as feature reuse, data transformations, and consistency between feature engineering, model training, and model inference. Columnar storage organizes data by columns, which is efficient for analytical queries that access a few attributes across many rows~\cite{jim_dowling_building_2025}, while row-oriented storage organizes data by rows, which is better suited for transactional operations that access all attributes of individual records~\cite{jim_dowling_building_2025}.

The platform provides centralized access to feature data and organizes \gls{ML} systems around three types of pipelines. The feature pipeline transforms input data into features stored in the feature store. The training pipeline reads features and labels to train models and registers them in a model registry. The inference pipeline reads new features alongside a trained model to produce predictions and prediction logs. Features are organized into feature groups, which are mutable tables of precomputed features, and feature views, which define the schema, filters, and helper columns for a given model. Feature computation is supported in three modes. Batch computation processes large volumes of data at scheduled intervals, generating features for subsequent training or inference. Streaming computation continuously processes incoming data in real time, updating features as new events arrive. Request-time computation calculates features on-demand at the moment a prediction is requested, ensuring that the model has access to the most up-to-date information. The platform also distinguishes between model-independent, model-dependent, and request-time features, enabling reuse across multiple models. External data sources can be integrated through services such as BigQuery, which is a serverless data warehouse for large-scale analytics~\cite{fernandes_what_2015}, Snowflake, a cloud-based platform for data storage and analytics~\cite{snowflake_inc_snowflake_2026}, \Gls{Amazon S3}, a scalable object storage service for storing and retrieving large datasets~\cite{armbrust_view_2010}, HopsFS, a distributed file system designed for high-performance ML workloads~\cite{niazi_hopsfs_2017}, RonDB, a distributed in-memory database optimized for real-time operations~\cite{hopsworks_ab_rondb_2026}, and OpenSearch, a search and analytics engine for querying and visualizing structured and unstructured data~\cite{lf_projects_llc_opensearch_2026}.

Having established how \gls{ML} models are engineered, operationalized, and supported by platforms, the following section examines \gls{DL}, the subset of \gls{ML} that underpins modern generative models.

\subsection{Deep Learning}

\Gls{DL} is a subset of \gls{ML} that can automatically learn hierarchical representations from raw data, reducing the need for manual feature engineering~\cite{lecun_deep_2015}. The term ``deep'' refers to the use of multiple successive layers of representation, in contrast to ``shallow'' models such as decision trees or support vector machines that do not learn through layered transformations. Modern deep networks commonly consist of hundreds or even thousands of layers, with architectures such as ResNet demonstrating that networks exceeding 1,000 layers can be trained effectively~\cite{he_deep_2016}. A typical \gls{DL} model consists of an input layer, one or more hidden layers, and an output layer. Each layer is made up of interconnected nodes, where each connection has an associated weight, forming a neural network. Each node computes a weighted sum of its inputs and then produces an activation, which is the output value passed to the next layer. A common non-linear activation function is the \gls{ReLU}, which outputs the input if it is positive and zero otherwise. By applying \gls{ReLU}, the network can bend and shape its outputs to approximate complex functions in the data, such as detecting edges in an image or predicting patterns in weather measurements.

During model training, the error quantifies the difference between the model's predictions and the desired outputs~\cite{schmidhuber_deep_2015}. The goal is to minimize this error by adjusting the network's weights. Gradients, which indicate how the error changes with respect to each weight, are computed and used to update the weights in a direction that reduces the error. A common optimization method is \gls{SGD}, where the model processes small batches of training examples, computes outputs and errors, and updates weights based on the average gradient over the batch. To efficiently compute gradients across multiple layers, backpropagation is used. Backpropagation applies the chain rule of calculus to propagate the error from the output layer backward through the network, allowing gradients for all weights to be calculated efficiently~\cite{he_deep_2016}.

The most prominent types of \gls{DL} networks include \glspl{RvNN}, \glspl{RNN}, and \glspl{CNN}~\cite{alzubaidi_review_2021}. \Glspl{RvNN} process structured data as trees or graphs. They are particularly useful in the context of \gls{NLP} for tasks where the hierarchical structure of a sentence matters. For example, a sentence can be split into individual words, and the model recursively combines pairs of words or phrases to form higher-level representations. At each step, the \gls{RvNN} computes a score for each pair of nodes, representing how plausible it is to merge them syntactically or semantically. The pair with the highest score is merged, and this process continues until a full tree representing the sentence structure is constructed. This allows the model to capture hierarchical relationships and compositional meaning in natural language.

\Glspl{RNN} are also widely used in \gls{NLP} as they process sequential data, where the order of inputs carries contextual information~\cite{alzubaidi_review_2021}. However, \glspl{RNN} are sensitive to the exploding gradient and vanishing gradient problems, caused by the repeated multiplication of large or small values during training. The vanishing gradient problem occurs when gradients become increasingly small as they propagate backward through many layers, preventing the network from learning long-range dependencies, while the exploding gradient problem occurs when gradients grow uncontrollably large, causing unstable weight updates. As a result, the network's ability to retain information from earlier inputs decays over time~\cite{bengio_learning_1994}. To address this, a specialized variant called \gls{LSTM} was introduced, which uses recurrent connections in memory blocks containing memory cells that store the temporal state of the network, as well as gated units that control the flow of information~\cite{hochreiter_long_1997}.

\Glspl{CNN} are a third type of \gls{DL} network~\cite{alzubaidi_review_2021}. Their structure is inspired by neurons in human and animal brains, and they are widely used in fields such as image recognition and speech processing. Unlike fully connected networks, \glspl{CNN} use shared weights and local connections to take advantage of the spatial structure in input data. This means only a small number of parameters are needed, which simplifies the training process and speeds up the network. A typical \gls{CNN} consists of multiple convolutional layers followed by pooling layers, with fully connected layers at the end.

In addition to architectures such as \glspl{CNN}, which are mainly applied to analyzing and classifying data, \gls{DL} also encompasses models designed to generate new data. This subset of approaches is commonly referred to as \gls{GenAI}.

\subsection{Generative Artificial Intelligence}

\Gls{GenAI} is defined as “the production of previously unseen synthetic content, in any form and to support any task, through generative modeling”~\cite{garcia_penalvo_what_2023}. Early generative models relied on probabilistic modeling with handcrafted probability distributions to generate synthetic data, but lacked the expressiveness to handle high-dimensional data, such as images, video, or genomic sequences. Early implementations included \glspl{RBM}, which are shallow two-layer networks where one layer represents visible inputs and the other captures latent features~\cite{hinton_fast_2006}. Latent features are hidden variables that summarize underlying patterns or structures in the data not directly observable in the input~\cite{jim_dowling_building_2025}. \Glspl{DBN} stack multiple \glspl{RBM} to learn progressively more abstract representations~\cite{schmidhuber_deep_2015}. While capable of unsupervised feature learning, both relied on Markov Chain Monte Carlo sampling, a method for approximating probability distributions through sequences of random samples~\cite{metropolis_equation_1953}, which limited their scalability~\cite{trigka_evolution_2025}.

A shift toward techniques based on neural networks followed, making \gls{GenAI} a subtype of \gls{DL}. \Glspl{VAE} introduced a probabilistic approach using an encoder-decoder architecture, where an encoder compresses input data into a compact latent representation and a decoder reconstructs the output from that representation~\cite{kingma_auto-encoding_2013}. This allowed \glspl{VAE} to model probability distributions in latent space, though they tended to produce blurry samples, limiting their applicability in high-fidelity image synthesis. \Glspl{GAN} took a different approach, using two neural networks, a generator and a discriminator, competing in an adversarial process. While \glspl{GAN} achieved significant improvements in output quality, they suffered from mode collapse, training instability, and sensitivity to hyperparameters, which are configuration settings such as learning rate and batch size that are set before training and influence how the model learns~\cite{jim_dowling_building_2025}. Subsequent variants addressed these limitations: deep convolutional \glspl{GAN} incorporated convolutional layers to improve the quality and stability of image generation, while Wasserstein \glspl{GAN} replaced the original loss function with the Earth Mover's distance, a metric that measures the minimum cost of transforming one probability distribution into another~\cite{rubner_earth_2000}, which helped stabilize training~\cite{trigka_evolution_2025}.

In parallel, the field of \gls{NLP} underwent significant developments. The introduction of the transformer architecture marked a turning point by replacing recurrent mechanisms, where inputs are processed sequentially and each output depends on the previous step, with self-attention, a mechanism that allows each element in a sequence to directly attend to all other elements simultaneously~\cite{vaswani_attention_2017}. This change enabled parallelized training over entire sequences rather than step-by-step processing. A transformer architecture is composed of stacked encoder or decoder blocks, where each block contains attention layers followed by feed-forward networks. An attention layer produces context-aware representations by allowing each element in a sequence to assign weights to all other elements and combine their information accordingly. A feed-forward network is a neural network layer that transforms each input independently using linear layers followed by a nonlinear activation~\cite{huang_da-code_2024}. Encoder blocks map the input sequence into a set of representations that capture both the meaning of each element and its relationships to other elements in the sequence. These representations are refined across multiple layers of attention and transformation. Decoder blocks generate the output sequentially. At each step, the decoder first applies self-attention over previously generated outputs, and then applies cross-attention over the encoder’s full input representation. Not all transformer models use both components. Encoder-only architectures such as \gls{BERT}~\cite{devlin_bert_2019} are designed for understanding tasks, decoder-only architectures such as \gls{GPT}~\cite{radford_improving_2018} focus on text generation, and encoder-decoder architectures such as T5~\cite{raffel_exploring_2023} combine both for tasks that require input understanding and output generation. These models rely on large-scale pre-training on extensive text corpora and are collectively referred to as \glspl{LLM}~\cite{bandi_power_2023}.

\subsubsection{Large Language Models}

\Glspl{LLM} are models capable of executing a diverse range of \gls{NLP} operations~\cite{raiaan_review_2024}. The most important concepts underlying an \gls{LLM} include tokenization, attention mechanisms, activation functions, normalization layers, training methods and frameworks, data pre-processing, and parameter tuning. Modern \glspl{LLM} can have \glspl{B} of parameters, which are the learned weights that determine the model’s ability to capture patterns in language~\cite{brown_language_2020}.

Tokenization refers to the process of parsing text into discrete units called tokens, which can be characters, sub-words, symbols, or words. In the attention mechanism, each token generates three vectors: a query, representing what information the token is looking for, a key, representing what information the token contains, and a value, representing the actual content to be passed forward. Attention scores are computed by comparing each query against all keys, and the output is a weighted sum of the corresponding values. Activation functions such as \gls{ReLU} introduce non-linearity, while normalization layers maintain stability during training by scaling and shifting activations so that they have a consistent mean and variance, which helps prevent exploding or vanishing gradients. Training at scale is enabled through parallelism strategies including data parallelism, pipeline parallelism, tensor parallelism, and optimizer parallelism. Data pre-processing involves quality filtering and data de-duplication. Parameter tuning techniques such as prefix tuning, prompt tuning, and adapter tuning allow models to be adapted to specific tasks without retraining all parameters. Beyond these architectural foundations, \glspl{LLM} can also be improved through human feedback.

\Gls{RL} from human feedback consists of three stages~\cite{wu_brief_2023}. First, supervised fine-tuning trains the model on prompts paired with desired responses, which can include plain prompts, few-shot examples with multiple query-response pairs, or user-based prompts for specific use cases. Second, a reward model is trained that takes a prompt and response as input and produces a scalar reward score. Third, the model is optimized using a \gls{RL} algorithm such as proximal policy optimization, where the model generates responses to prompts and receives reward scores that guide further training~\cite{schulman_proximal_2017}.

In addition to training-time improvements, \glspl{LLM} exhibit in-context learning, a meta-learning method in which the model uses contextual information provided in a prompt to solve specific tasks without updating its weights~\cite{yin_survey_2024}. In-context learning takes several forms. Zero-shot learning provides only a task description, while few-shot learning includes examples within the prompt, allowing the model to learn by analogy. Chain-of-thought prompting extends this by constructing intermediate reasoning steps that simulate human thinking, improving performance on tasks requiring arithmetic, commonsense, and logical reasoning. Instruction fine-tuning is a further form, where models are trained on datasets of instructions paired with desired outputs. However, all of these techniques are constrained by the context window.

The context window defines the maximum number of tokens an \gls{LLM} can process in a single forward pass~\cite{ratner_parallel_2023}. A forward pass refers to the process in which input data flows through the neural network's layers to produce an output~\cite{schmidhuber_deep_2015}, such as predicting the next token. The computational cost of this operation scales quadratically with context length due to the self-attention mechanism, which computes relationships between every pair of tokens in the input sequence. This trade-off must be considered when processing large inputs, as model performance can become increasingly unreliable as the input length grows, a phenomenon referred to as ``context rot'' in industry~\cite{hong_context_2025}.

To make effective use of this limited context window, prompt engineering emerged as ``the practice of designing and refining input prompts to achieve desired outputs from generative \gls{AI} models''~\cite{cinar_prompt_2026}. Prompt engineering extends in-context learning techniques described above to maximise model performance without adjusting model weights. Over time, entire design philosophies have been developed around prompt construction, such as a problem-oriented, case-driven, and knowledge-complete approach~\cite{wu_improving_2025}. Problem-oriented means that context is established based on questions posed by technical personnel in real-world scenarios, ensuring that the model's understanding aligns with actual work contexts. Case-driven means including fault cases for the model to learn from, and adding technical terminology and principles as knowledge points. These problem descriptions and fault cases precede the actual instruction. As a result, accuracy in executing tasks improves by up to 30\%.

While \glspl{LLM} benefit from effective prompt engineering and in-context learning, they also encode and propagate societal biases present in their training data~\cite{gao_detecting_2026}. These biases manifest in two principal forms. Representational harms arise when a model reinforces stereotypes or misrepresents groups, for example by systematically completing ``The nurse said that\ldots'' with male pronouns despite real-world demographics. Allocational harms occur when model outputs lead to unequal treatment, such as toxicity classifiers producing higher false-positive rates for posts written in specific dialects, resulting in disproportionate content removal. Detecting and measuring such biases requires a layered evaluation approach combining intrinsic methods that probe embeddings and likelihoods, extrinsic methods that assess output-level disparities across tasks, and certification-based techniques that provide statistical guarantees on fairness metrics. Persistent challenges remain, including evaluator bias, limited coverage of non-English languages and intersectional groups, and the risk of over-correction where models achieve superficial fairness by refusing to engage with sensitive topics rather than responding equitably.

Despite these biases and the ongoing challenges in their detection and mitigation, the general-purpose capabilities of \glspl{LLM} have made them particularly attractive for code-related tasks in \gls{SWE}, leading to a growing number of models specifically adapted or built for code generation.

\subsubsection{Code Generation Models}

The recent surge in the development of \glspl{LLM} has led to a significant number of these models being repurposed for code generation through continual post-training, where a model already trained on general text is further trained on large code corpora, or fine-tuning, where a pre-trained model is adapted to a specific task using a smaller, targeted dataset~\cite{jiang_survey_2026}. This trend is particularly observable in open-source models, which are AI models whose architecture, weights, and training code are publicly available~\cite{touvron_llama_2023}, allowing researchers and developers to study, modify, and deploy them freely. In this context, general-purpose models have been tuned into code-specific variants: \gls{LLaMA} into Code \gls{LLaMA}, DeepSeek \gls{LLM} into DeepSeek Coder, and Qwen into Code Qwen. On the closed-source side, models such as Claude Opus 4.6~\cite{anthropic_pbc_system_2026} and \gls{GPT}-5.3-Codex~\cite{openai_gpt-53-codex_2026} represent proprietary offerings with strong code generation capabilities. Beyond adapting general-purpose \glspl{LLM}, other models have been engineered specifically for code generation from the outset.

Coding models exist within a broader ecosystem that can be understood as a dynamic network of \glspl{LLM} and datasets focused on \gls{SWE} tasks~\cite{yang_ecosystem_2026}. Model hubs such as HuggingFace play a central role, providing platforms where users and companies upload, share, and train models. Within this ecosystem, models interact through complementary and derivative relationships, often evolving from foundational models into specialized variants through reuse strategies such as quantization, which reduces model size and computational cost by lowering the precision of numerical weights, and adversarial training, which exposes the model to deliberately challenging or misleading inputs during training to improve its robustness. Company and non-company users contribute a similar number of models, though models from company users receive significantly more engagement, and notably, a model's popularity does not always align with its importance in the ecosystem. The ecosystem is dominated by less restrictive licenses such as Apache-2.0, which permits commercial use while requiring attribution~\cite{the_apache_software_foundation_apache_2026}, \gls{BSD}-3-Clause, which allows broad redistribution with minimal restrictions~\cite{opensourceorg_3-clause_2026}, and \gls{MIT}, which imposes virtually no conditions beyond preserving the license notice~\cite{opensourceorg_mit_2026}. While no license incompatibility issues have been identified so far, potential risks remain. A persistent challenge is that the metadata and documentation of code generation models is generally less informative than that of traditional \gls{AI} repositories on GitHub, calling for greater attention to documentation quality.

The capabilities of code generation models are commonly evaluated using \gls{SWE}-bench, a standardised benchmark that measures a model's ability to resolve real-world \gls{SWE} tasks~\cite{jimenez_swe-bench_2024}. The benchmark consists of 2,294 \gls{SWE} problems drawn from real GitHub issues and corresponding pull requests across 12 popular Python repositories. Given a codebase and an issue description, a model is tasked with generating a patch that resolves the issue, with success determined by whether the patched codebase passes the associated test suite. Resolving these issues frequently requires understanding and coordinating changes across multiple functions, classes, and files simultaneously, making \gls{SWE}-bench a challenging testbed that goes far beyond traditional code generation tasks. Early evaluations showed that the best-performing model at the time, Claude 2, resolved only 1.96\% of issues. Since then, progress has been immense. Claude Opus 4.6 achieves 80.8\% and \gls{GPT}-5.2 by \gls{OpenAI} achieves 80\% on the \gls{SWE}-bench Verified subset~\cite{anthropic_pbc_system_2026}.

Despite their growing capabilities, code generation models are prone to several categories of errors, including misinterpretation of prompts, syntax errors, hallucinated objects or attributes, missing corner cases, incomplete generation, and prompt-biased code where the model excessively relies on provided examples~\cite{tambon_bugs_2025}. To mitigate these issues, providing additional context such example test cases or explicit library versions guides the model toward more accurate output. Users should also test generated code, and remain mindful that it may not always be reliable.

Empirical comparisons reinforce these concerns. When compared to search-based software testing tools such as EvoSuite~\cite{siddiq_using_2024}, \glspl{LLM} performed worse across all studied dimensions, in part because they do not always produce compilable unit tests. Models fine-tuned for code generation, such as Codex and StarCoder, outperformed general-purpose models like \gls{GPT}-3.5-Turbo, and hallucination of nonexistent types and methods was the most frequent compilation error observed.

Another concern is robustness, the ability of a model to produce correct output when faced with semantically equivalent but syntactically varied inputs. A study evaluating 25 popular \glspl{LLM} across four programming languages and 27 semantics-preserving code transformations found that current models still exhibit notable robustness issues, although they outperform earlier pre-\gls{ChatGPT} code models~\cite{liu_robustness_2026}. The study revealed that instruction-tuned models are generally more robust than their base counterparts, that higher overall performance correlates with smaller performance drops under transformation, and that models are particularly sensitive to transformations altering program control structure. While techniques such as adversarial training, in-context learning, and code standardization were shown to improve robustness to some extent, each also carried potential negative side-effects, indicating that no single mitigation strategy is yet sufficient on its own.

Nonetheless, \glspl{LLM} are increasingly integrated into software development workflows through the generation of initial code artifacts from documentation, which can reduce effort in early-stage development, although subsequent manual refinement by developers remains necessary. These models and their capabilities form the foundation for emerging development practices such as vibe coding.

\subsubsection{Vibe Coding}

\Glspl{LLM} are increasingly applied to \gls{SWE} tasks, spanning requirements engineering, software design, development, quality assurance, maintenance, and management~\cite{hou_large_2024}. Decoder-only architectures are the most widely adopted, and code generation and program repair are the most prevalent tasks within software development and maintenance activities.

With the increasing popularity of using \glspl{LLM} for code generation, a new term emerged to describe the practice. Vibe coding was coined in February 2025 by Andrej Karpathy, who described it as a ``new kind of coding where you fully give in to the vibes, embrace exponentials, and forget that the code even exists''~\cite{andrej_karpathy_karpathy_theres_2025}. More formally, vibe coding is defined as ``development practices that foster intuition, expression, and human-AI collaboration''~\cite{beaulieu_vibe_2025}. Four primary themes shape the practice. Tooling impact concerns tools, workflows, and mindsets related to AI-assisted coding. Developer experience frames software development as intuitive, expressive, and flow-based. Human-AI collaboration highlights interactions such as prompt iteration, co-creative workflows, and task delegation. Cultural and organizational framing examines how vibe coding influences team structures and organizational practices.

Three architectural patterns have been identified in vibe coding~\cite{akhilesh_gadde_democratizing_2025}. Firstly, conversational code generation involves a natural language prompt followed by \gls{LLM} processing, code generation, and user feedback. Secondly, multimodal expression processing accepts inputs beyond text, such as sketches, and fuses them into a unified intent model before generating code. Thirdly, context-aware generation analyzes the existing codebase, extracts architectural patterns, interprets user intent, and produces code that aligns with and integrates into the existing system.

Interaction patterns between the user and the \gls{LLM} also vary~\cite{akhilesh_gadde_democratizing_2025}. Progressive refinement starts with a high-level description, generates initial code, and iteratively adds constraints and specifications to produce improved versions. Explanation-driven development involves the user elucidating generated code, followed by conceptual analysis and refinement. Hybrid editing combines natural language guidance with direct code review, allowing the user to alternate between describing intent and inspecting implementation.

A prominent application of vibe coding is in software prototyping, where it substantially lowers the barrier to create high-fidelity executable prototypes~\cite{borg_vibe_2025}. Traditional prototyping ranges from low-fidelity artifacts such as sketches and wireframes to high-fidelity executable software, with the latter requiring significant engineering competence. Vibe coding reduces this requirement by enabling non-technical users to produce interactive, executable prototypes through natural language interaction with platforms such as Lovable. Five primary purposes of prototyping benefit from this shift: external communication with customers and investors, internal alignment across development teams, validation and usability testing, incremental development, and exploration of product-market fit. By reducing turnaround times and dependence on traditional development skills, vibe coding facilitates rapid parallel experimentation, including A/B testing and early usability evaluation. A/B testing is a controlled experimental method in which two or more variants of a system are compared by randomly assigning users or observations to each variant and measuring differences in predefined performance metrics to determine which performs better~\cite{shahriari_taking_2016}. However, qualities essential for evolutionary prototyping, such as maintainability, security, and performance, remain open concerns when development prioritizes speed and creative flow over engineering rigour.

While vibe coding leverages \glspl{LLM} to automate software development through human-\gls{AI} collaboration, a parallel line of research increasingly explores the use of \glspl{LLM} and code generation models within \gls{AutoML}, where the goal extends from generating individual code snippets to generating complete \gls{ML} pipelines.

\subsection{Automated Machine Learning}

\gls{AutoML} is defined as ``the automated construction of an \gls{ML} pipeline on a limited computational budget''~\cite{he_automl_2021}. A complete \gls{AutoML} pipeline as illustrated in Figure~\ref{fig:automl} consists of four sequential stages. The first stage, data preparation, encompasses data collection, data cleaning, and data augmentation, ensuring that the raw data is sourced, filtered of noise and inconsistencies, and artificially expanded to improve model robustness. The second stage, feature engineering, transforms the cleaned data into a refined feature set through three complementary processes. Feature selection removes irrelevant or redundant attributes. Feature extraction applies mapping functions to create more compact representations. Feature construction generates new features by combining existing ones. The third stage, model generation, is divided into the search space and the optimization methods. The search space defines the models that can be explored, ranging from traditional \gls{ML} models to deep neural networks. The optimization methods are classified into hyperparameter optimization and architecture optimization. Architecture optimization is responsible for tuning model-related parameters such as the number of layers or neurons. When the search space is restricted to deep neural networks and combined with architecture optimization, this sub-field is referred to as \gls{NAS}. The fourth stage, model evaluation, determines how well a generated model performs. Several acceleration strategies are used to reduce the computational cost of training every candidate to full convergence. These include low-fidelity evaluation, early stopping, which halts training when performance on a validation set ceases to improve~\cite{montavon_early_2012}, surrogate models, which approximate the objective function to reduce the number of expensive evaluations~\cite{queipo_surrogate-based_2005}, and weight sharing, which allows candidate architectures to reuse learned parameters rather than training from scratch~\cite{lecun_gradient-based_1998}. Throughout the pipeline, model generation and model evaluation operate in an iterative feedback loop: candidate models are generated, evaluated, and refined until the best-performing pipeline is identified.

    \caption{Overview of an \gls{AutoML} pipeline~\cite{he_automl_2021}}
    \label{fig:automl}
\end{figure}

The integration of \glspl{LLM} into \gls{AutoML} has led to a new generation of \gls{ML} systems that leverage natural language understanding to reduce manual effort across the \gls{ML} pipeline. An early example is Aliro, a platform that combines \glspl{LLM} with a built-in recommender system for the selection of \gls{ML} algorithms and hyperparameters~\cite{choi_aliro_2023}. Aliro supports chat-based interaction, code editing, package installation, data visualization, and data processing. Users provide an \gls{OpenAI} \gls{API} key to enable \gls{LLM}-driven code generation, and the system offers the ability to save artifacts produced by scripts, such as text and image files. Its key contribution is the \gls{LLM}-based recommender system, which guides users through model building without requiring deep expertise, as demonstrated in a use case involving Alzheimer's disease research using a random forest classifier, a \gls{ML} algorithm used for classification tasks~\cite{goodfellow_deep_2016}.

Building on this direction, \gls{LLM2AutoML} proposed a zero-code \gls{AutoML} framework that integrates \glspl{LLM} across intention understanding, model recommendation, and hyperparameter optimization~\cite{chen_llm2automl_2024}. The framework employs prompt engineering for intention extraction, in which the \gls{LLM} assumes the role of an experienced \gls{ML} expert, with prompts refined through iterative feedback cycles. Key capabilities include massive feature extraction with recursive feature elimination to identify salient features, meaning those that contribute most strongly to the predictive performance of a model~\cite{jim_dowling_building_2025}, as well as automatic ablation studies and automated hyperparameter optimization. The framework also introduces adaptive focal loss as an alternative to traditional cross-entropy loss, a loss function that measures the divergence between predicted probability distributions and actual class labels and is widely used as the default objective in classification tasks~\cite{rumelhart_learning_1986}. Report generation is also automated. The framework was evaluated on the \gls{SECOM} dataset to emulate semiconductor smart manufacturing, comparing \gls{OpenAI} o1-mini and Claude-3.5-Sonnet. Results showed that \gls{LLM}-recommended hyperparameter spaces improved model performance, particularly in convergence speed and \gls{AUC} values, a metric that quantifies how well a classifier distinguishes between classes by measuring the area under the receiver operating characteristic curve~\cite{alzubaidi_review_2021}. This demonstrated strong generalization capabilities and expert-level tuning performance when integrated with search algorithms.

A systematic survey then provided the first \gls{SWE}-oriented, stage-wise mapping of \gls{LLM} contributions across the full \gls{AutoML} workflow, analyzing 34 studies published after 2022~\cite{gu_large_2025}. The survey organizes contributions into three stages. In the first stage, data and feature engineering, \glspl{LLM} support data acquisition through dataset recommendation and summarization. For data cleaning, they enable adaptive imputation, which dynamically selects strategies for filling in missing values based on contextual cues in the data, and context-enhanced transformation, which uses surrounding semantic information to inform how data values are reformatted or standardized. Data augmentation is supported through both modification of existing samples and synthetic data generation. For feature engineering, \glspl{LLM} are employed across feature selection, extraction, and synthesis, with frameworks integrating these into pipelines that enable dynamic operator composition, the automated selection and chaining of feature transformation operations at runtime. In the second stage, model selection and hyperparameter optimization, retrieval-based methods match task descriptions against curated model repositories to recommend candidate algorithms, while generation-based methods synthesize selection logic or entire architectures directly from natural language. For hyperparameter optimization, execution-based methods iteratively refine configurations by training models and observing real performance feedback, while prediction-based methods bypass full training by using surrogate models to simulate outcomes. In the third stage, workflow evaluation, agent-based methods deploy \gls{LLM}-powered agents that predict performance from task descriptions or partial executions without completing full training cycles, while proxy-based methods employ lightweight surrogate models or zero-cost heuristics, approximation strategies that estimate metrics without exhaustive computation~\cite{hart_formal_1968}, to approximate key metrics in a single forward pass.

Across all stages, the survey identifies persistent challenges around transparency, correctness, and robustness, and highlights that production-grade adoption demands maintainability, interpretability, and traceability of generated logic. Among its future research directions, the survey calls for embedding \gls{ML} workflow generation into full-stack data applications. It argues that workflows should integrate upstream with data lakes, catalogs, and schema managers to guarantee provenance and compliance, and downstream with dashboards, visualization layers, and application-specific front-ends to enable seamless enterprise decision-making. A data lake is a centralized repository that stores large volumes of raw, structured, semi-structured, and unstructured data in its native format, allowing for flexible analytics and processing~\cite{hai_data_2023}. This would allow \gls{LLM}-generated \gls{AutoML} workflows to evolve from research prototypes into production-grade systems. Beyond standalone \glspl{LLM}, \gls{LLM}-based \gls{AI} agents equipped with specialized skills and tools are increasingly employed to automate such workflows. The following section introduces the concept of agents and their underlying foundations.

\section{Agents}
\glsresetall

The concept of agents has a long history in \gls{AI}, originating in research on rational decision-making and autonomous behavior. Early work formalized how individual agents perceive their environment, form beliefs, and act on intentions. As these ideas matured, attention shifted toward systems of multiple agents operating within shared environments, giving rise to \gls{DAI} and multi-agent systems concerned with coordination, planning, and strategic interaction.

The translation of these theoretical foundations into software led to software agents, autonomous programs designed to operate continuously within computational environments. While early software agents were generally fragile and special-purpose, the convergence of agent architectures with \glspl{LLM} has produced a new generation of coding agents capable of interpreting developer goals, generating code, and interacting with external tools. As these agents grow more capable, their effectiveness increasingly depends on the context they receive and how efficiently it is managed, motivating the examination of context engineering in the following chapter.

\subsection{Intelligent Agents}

Agents originated in research on rational decision-making and autonomous behavior. A rational agent’s behavior is governed by three mental attitudes, known as the \gls{BDI} model~\cite{rao_modeling_1991}. Beliefs represent the set of worlds the agent considers possible at a given point in time, referred to as belief-accessible worlds. Desires represent the set of worlds the agent would like to bring about, known as goal-accessible worlds. Intentions are the subset of desires that the agent has committed to realizing, giving rise to intention-accessible worlds. Decision-making emerges from comparing belief-accessible worlds and selecting goals that are consistent with these beliefs, such that for every belief-accessible world there exists a corresponding goal-accessible world, although the converse does not necessarily hold. Building on this formalization of rational agents, the concept can be further specified in terms of the observable properties that characterize intelligent agents in practice.

An agent is defined as ``a hardware or (more usually) software-based computer system that enjoys the following properties: autonomy [...], social ability [...], reactivity [...], and proactiveness''~\cite{wooldridge_intelligent_1995}. Autonomy is the ability to operate without direct human interaction. Social ability is the ability to interact with other agents. Reactivity is the ability to perceive the environment and respond to changes that occur in it. Proactiveness means that agents do not simply respond to their environment but exhibit goal-directed behavior. Agents can also be described as intentional systems, where a first-order intentional system has beliefs and desires, but no beliefs and desires about beliefs and desires. A second-order intentional system has beliefs and desires about beliefs and desires.

Early applications of intelligent agents spanned a range of domains~\cite{jennings_applications_1998}. In manufacturing and logistics, agents were applied to traffic management, flexible manufacturing, and plant control using modular architectures based on behavior, resources, and intentions. In telecommunications, agent-based systems addressed network control, service management, transmission switching, and network design. On the internet, agents were deployed for filtering, evaluating, and integrating information, often operating in teams depending on user tasks and situational knowledge. Further applications emerged in knowledge discovery and information analysis across large data spaces, as well as in financial services, including mortgage sales, corporate financial management, and portfolio management.

While the properties of autonomy, reactivity, and proactiveness characterize individual agent behavior, many of the applications described above involve multiple agents interacting within a shared environment, so called multi-agent systems studied in the field of \gls{DAI}.

\subsection{Distributed Artificial Intelligence}

\Gls{AI} has traditionally focused on the capabilities of individual systems, but many real-world problems are inherently distributed in nature, involving multiple entities that must operate across different locations, manage separate sources of information, and pursue potentially conflicting objectives~\cite{panait_cooperative_2005}. \Gls{DAI} emerged as a field to address these challenges by combining principles from \gls{AI} with distributed computing. \Gls{DAI} encompasses two main areas: distributed problem solving, which involves the decomposition and distribution of a problem-solving process among subordinate nodes, and multi-agent systems, which focuses on the joint behavior of agents with some degree of autonomy.

A multi-agent system consists of multiple individual agents that interact with each other through communication and can act within a shared environment~\cite{wooldridge_introduction_2012}. Different agents may have different spheres of influence and control over different parts of the environment. Various relationships can exist between agents, including coincidence, dependency, power relationships, and hierarchies. Agents are assumed to be self-interested, possessing their own preferences and desires. More specifically, dependency relationships between agents can take several forms: independent, where no dependence exists between agents, unilateral, where one agent depends on another, mutual, where both agents depend on each other with respect to the same goal, and reciprocal, where the first agent depends on the second for one goal and the second depends on the first for another.

These relationships shape how agents reason about outcomes and make decisions~\cite{wooldridge_introduction_2012}. The set of possible outcomes or states over which agents have preferences is evaluated through a utility function, which provides a formal way of representing an agent's preferences. The actual outcome depends on the combination of actions performed by all agents. A dominant strategy is one that guarantees an agent the best possible payoff regardless of what other agents do. A Nash equilibrium is a state in which neither agent has an incentive to unilaterally deviate from its current strategy. In zero-sum interactions, the outcome gains of one agent are exactly offset by the losses of another.

Coordination in multi-agent systems requires a structure within which agents can interact in predictable ways, flexibility so that agents can operate in dynamic environments, and appropriate knowledge and reasoning capabilities to intelligently use both the structure and the flexibility~\cite{jennings_commitments_1993}. Commitments produce predictability, which agents need to reason about, while conventions allow agents to respond flexibly to changing circumstances. Coordination can thus be understood as a distributed goal search problem.

Closely related to coordination is the problem of planning, which determines how agents structure their actions toward shared objectives~\cite{grosz_collaborative_1996}. Planning in multi-agent systems involves both individual plans and shared plans. A shared plan for group action incorporates individual and mutual beliefs, and can take several forms: full individual plans, partial individual plans, full shared plans, partial shared plans, and shared plans of indefinite completeness. A collaborative plan for action includes a mutual belief of a partial recipe for the action, individual intentions that the action be done, individual intentions that collaboration succeed in performing the constituent sub-actions, and individual or collaborative plans for the sub-actions.

While the concepts of coordination, planning, and strategic interaction provided a robust theoretical foundation for multi-agent systems, translating these ideas into working software required addressing practical concerns such as continuous operation, communication standards, and integration with existing computing infrastructure. This led to the emergence of software agents as a distinct area of research.

\subsection{Software Agents}

With the emergence of intelligent agents, research also focused on how to integrate agents into \gls{SWE} processes. A software agent is defined as ``a software entity which functions continuously and autonomously in a particular environment, often inhabited by other agents and processes''~\cite{bradshaw_introduction_1997}. Such an agent can operate continuously over a long period of time, learn from experience, and coexist with other agents and processes within a computer system.

Distinguishing software agents from other agent types, one taxonomy categorizes autonomous agents into biological, robotic, and computational agents, with computational agents further divided into software agents and artificial life agents. Software agents are further classified into task-specific agents, entertainment agents, and viruses~\cite{franklin_is_1997}. An alternative, more extensive classification identifies seven types of software agents: collaborative agents, interface agents, mobile agents, information agents, reactive agents, hybrid agents, and heterogeneous agent systems~\cite{nwana_software_1996}. Collaborative agents emphasize autonomy and cooperation with other agents to solve problems too large for a single agent. Interface agents emphasize autonomy and learning to perform tasks on behalf of their owners, comparable to a personal assistant, by observing and imitating the user, receiving positive and negative feedback, receiving explicit instructions, and consulting other agents for advice. Reactive agents do not possess internal models of their environment, instead acting in a stimulus-response manner without a priori specification of behavior. Hybrid agents combine philosophies from two or more agent types, while heterogeneous agent systems integrate at least two agents belonging to two or more distinct agent classes.

Building on these classifications, agent-based \gls{SWE} emerged as an approach to modelling, designing, and implementing computer systems using software agents~\cite{jennings_agent-based_2000}. The agent-oriented approach advocates decomposing problems in a way that allows agents to engage in flexible, high-level interactions. Each agent maintains its own thread of control and decides autonomously which actions to perform. This supports the engineering of complex distributed systems in several ways. Agent behavior can be based on the agent's actual local state rather than an external entity's perception of it. It is not necessary to know all potential interactions in advance. Agents are also designed to handle unanticipated requests and to spontaneously generate requests for assistance. If unexpected events occur, agents have the autonomy and proactiveness to try alternative actions. Because schedules are built up dynamically through flexible interactions, they can be readily adapted in the event of delays or unforeseen contingencies. Additionally, explicitly defined relationships between system components identify which agents need to coordinate their actions.

Alongside these developments, efforts were made to establish universal communication languages for agents. Two approaches emerged: a procedural approach, based on the exchange of sequential directives, and a declarative approach, based on the exchange of assertive statements~\cite{genesereth_software_1994}. The procedural approach was limited by its dependence on knowledge of the recipient and its unidirectional nature. The declarative approach gave rise to agent communication languages such as the \gls{KIF} and the \gls{KQML}.

However, agent communication languages failed to achieve widespread adoption and have largely fallen out of use. Early software agents were similarly constrained, generally fragile and special-purpose in nature~\cite{bradshaw_introduction_1997}. Software agents have nonetheless seen a resurgence in recent years. Today, coding agents combine the principles of intelligent agents with the ability to generate, execute, and maintain code using \glspl{LLM}.

\subsubsection{Coding Agents}

The convergence of \glspl{LLM} and software agents has led to a new generation of \gls{LLM}-based \gls{AI} agents~\cite{wang_survey_2024}, hereafter referred to as agents. These agents typically consist of four components as depicted in Figure~\ref{fig:agent}: planning, memory, perception, and action. Planning and memory serve as the \gls{LLM}-controlled brain, while the agent interacts with its environment through perception and action to achieve specific goals. The planning component decomposes complex tasks into subtasks and schedules them accordingly, generating or adjusting plans as needed. Memory records historical thoughts, actions, and environmental observations. Perception handles multi-modal inputs such as textual, visual, or auditory signals, and the action component executes concrete operations to interact with and influence the environment, including the control and use of external tools that extend beyond the inherent capabilities of \glspl{LLM}.

    \centering

    \caption{Basic framework of \gls{LLM}-based \gls{AI} agents~\cite{liu_large_2026}}

Coding agents are loosely defined as agents that, rather than merely executing predefined technical instructions, participate in interpreting a developer's explicit or implicit goals, anticipate potential needs, and provide personalized outputs based on usage patterns~\cite{meske_vibe_2025}. Most significantly, these agents are capable of offering solutions that may exceed the developer's own knowledge. Popular examples of coding agents include Claude Code~\cite{anthropic_pbc_claude_2026} and GitHub Copilot~\cite{github_inc_github_2026-1}.

Common use cases for coding agents include automated program repair, where traditional approaches rely on fault localization, semantic analysis, or validation~\cite{puvvadi_coding_2025}. Coding agents can also automate version management tasks such as upgrading dependencies, fixing compatibility issues, addressing deprecated \glspl{API}, and handling syntax changes. Additionally, coding agents support quality assurance and code review prior to human evaluation by suggesting optimizations, detecting inconsistencies, and validating compliance.

Early empirical evidence suggests that coding agents are already contributing meaningfully to real-world software projects~\cite{watanabe_use_2026}. Both agentic and human-written pull requests primarily address bug fixes and feature development. However, agentic pull requests more frequently involve refactoring, documentation updates, and test improvements, while human contributions tend to focus more on project-specific maintenance tasks such as continuous integration and dependency management. Agentic pull requests are also more likely to serve multiple purposes within a single submission, include more code additions, and contain longer descriptions. In terms of acceptance, agentic pull requests are widely adopted in real-world projects and achieve an 83.8\% acceptance rate, with review times comparable to those of human contributions, although still lower than the 91.0\% acceptance rate for human-written pull requests. Most rejections stem from project context, such as overlapping changes, verification-only submissions, or excessive size, rather than inherent flaws in the generated code. Nevertheless, technical challenges such as suboptimal design and unnecessary complexity, as well as trust-related concerns, remain. Among merged agentic pull requests, 45.1\% required reviewer revisions, primarily for bug fixes, documentation improvements, refactoring, and code style adjustments. This highlights that while \gls{AI}-generated code provides a strong starting point, human oversight remains essential to ensure correctness, maintainability, and adherence to project standards.

Multiple coding agents can also collaborate within a multi-agent system, where each agent assumes a specialized role in the development process. One example in this domain is automated dependency upgrading, implemented through a system comprising a summary agent, a control agent, and a code agent~\cite{tawosi_llm_2025}. The summary agent generates code summaries at varying levels of granularity. The control agent identifies relevant code units by first examining file-level summaries and then drilling down to finer-grained representations, while also consulting migration guides when available. The code agent receives the task along with the selected code units, implements the required changes, and updates dependencies and configurations. After each edit cycle, the project is compiled and tests are executed. If errors occur, logs are returned to the control agent for further resolution, forming an automated program repair loop. This process continues until the build succeeds or a maximum retry threshold is reached. At that point, the framework supports handover to human developers with a summary of the actions taken.

Beyond the technical architecture of these systems, the organizational relationship between human developers and coding agents is also evolving. One framework conceptualizes a human–agent team in which developers act as creative leads while agents serve as force multipliers~\cite{gulli_coding_2025}. This approach is grounded in three foundational principles. The first is human-led orchestration, whereby developers determine which agents to engage, provide context, and retain final decision-making authority. The second is the primacy of context. Agent performance depends heavily on the quality of the provided context, which should include the complete codebase, relevant external knowledge such as documentation and \gls{API} references, and a clear human-authored brief outlining goals, requirements, and pull request descriptions. The third principle is direct model access, ensuring that developers interact with the underlying model without unnecessary abstraction layers. Within this framework, the human developer acts as the orchestrator, using their preferred tools such as terminals, text-based interfaces for issuing commands to the operating system~\cite{sutherland_sketchpad_1963}, editors, or web interfaces. A context staging area allows the preparation of task-specific briefings for specialized agents, including a scaffolder agent for feature implementation, a test engineer agent for unit and integration testing, a documenter, an optimizer for performance improvements, and a process agent responsible for identifying potential bugs, style violations, and logical flaws. The developer retains architectural ownership, enforces quality standards, and engages in iterative dialogue with the agents, providing clear and comprehensive context for each task.

As coding agents continue to advance, their effectiveness increasingly depends not only on their internal reasoning capabilities but also on the context they receive and how efficiently it is managed within the constraints of the context window. This growing need to optimize what information is presented to agents, and how, motivates a closer examination of context engineering.

\subsubsection{Context Engineering}

Especially in domains that require large volumes of textual input, such as coding agents, techniques have been discovered extend in-context learning and prompt engineering by optimizing external context before adding it to the model~\cite{yusuf_beyond_2025}. For example, in code completion, providing top-k results to a coding agent after ranking source code using a keyword-based search algorithm such as \gls{BM25} can yield a 3\% improvement using Kotlin and a 9\% improvement using Python. File ordering also affects performance, as placing the most relevant files closest to the cursor position and the least relevant files at the beginning of the context ensures that when the context window is exceeded, the agent truncates from the left, discarding the least useful files first, resulting in a 3\% improvement for Python. Chunking strategies similarly influence results, as standard chunking ignores import statements and segments code into objects, classes, standalone functions, expressions, statements, and docstrings, whereas method-level chunking splits code into individual methods, again yielding a 3\% improvement when using Python. Lastly, local scope context trimming of prefix and suffix achieves a 3\% improvement in Kotlin.

With such techniques in mind and as \glspl{LLM} became the foundations of modern software agents, prompt engineering evolved into context engineering, defined as ``the architectural design of AI systems in which model behavior is shaped by dynamically constructed context beyond the immediate user prompt, including persistent memory, external knowledge retrieval, tool invocation, and policy enforcement mechanisms''~\cite{cinar_prompt_2026}. What distinguishes context engineering from prompt engineering is its focus on integrating external tools in addition to user requirements and the model's own perception, as well as using memory mechanisms to achieve continuity across interactions.

One of the most prominent mechanisms to achieve such continuity is \gls{RAG}. In a typical \gls{RAG} pipeline, documents are first segmented into passages, encoded as dense vector representations~\cite{fan_survey_2024}, and then stored in a vector database~\cite{ramos_review_2025}. When a query is received, a retriever encodes the query into the same embedding space and retrieves the top-k most similar passages using approximate nearest neighbour search. These passages are then concatenated with the original query into a prompt, and a \gls{LLM} generates a response conditioned on both the query and the retrieved context. By grounding responses in retrieved documents, \gls{RAG} reduces hallucinations~\cite{shuster_retrieval_2021}, model outputs ``that appear nonsensical or unfaithful to the provided source content''~\cite{huang_survey_2025}, and enables models to access knowledge that was not present during training.

Alternatives to \gls{RAG} promise increased accuracy in retrieving external context and reduced token cost. One such approach is Mem0, a memory framework that operates through two phases, namely extraction and update~\cite{chhikara_mem0_2025}. During extraction, salient facts are identified from new message pairs using a conversation summary and recent message history as context. During the update phase, each extracted fact is compared against existing memories using semantic similarity, and an \gls{LLM} determines whether to add, update, delete, or ignore it. In its base configuration, Mem0 stores memories as natural language text indexed by vector embeddings. An extended variant, Mem0\textsuperscript{g}, additionally represents memories as a directed labeled graph in which entities serve as nodes and relationships as edges, enabling relational reasoning across interconnected facts.

While context engineering focuses on supplying agents with the right information at the right time, it represents only one dimension of agent capability. Equally important is the ability to act on that context by interacting with external systems through tool use.

\subsection{Tools}

The ability of \glspl{LLM} to use external tools through \glspl{API} has emerged as a key research direction for overcoming inherent model limitations such as arithmetic errors and outdated factual knowledge. Toolformer was the first approach to demonstrate that tool use can be learned in a self-supervised way, without requiring large amounts of human annotations~\cite{schick_toolformer_2023}. The approach represents \gls{API} calls as linearized text sequences embedded within the token stream, delimited by special tokens that mark the start of a call, the arguments, the returned result, and the end of the call. At inference time, the model performs auto-regressive decoding, the standard process by which a language model generates text one token at a time, with each new token conditioned on all previously generated tokens~\cite{zhong_extractive_2020}. This proceeds normally until the model generates a special token indicating that it expects a response from an \gls{API}, at which point decoding is paused, the host system parses the \gls{API} name and arguments from the generated tokens, executes the call against the appropriate external tool, injects the result into the sequence, and resumes decoding. By incorporating tools such as a calculator, a question-answering system, a search engine, a translation system, and a calendar, Toolformer achieved substantially improved zero-shot performance across a variety of downstream tasks with a 6.7 billion parameter model, often reaching performance competitive with much larger models without sacrificing its core language modeling abilities.

Building on this foundation, Gorilla addressed the challenge that \glspl{LLM} remain largely unaware of which \glspl{API} are available and how to use them in a frequently updated tool set~\cite{gonzalez_gorilla_2024}. Gorilla is a fine-tuned \gls{LLaMA} model that surpasses the performance of \gls{GPT}-4 on writing \gls{API} calls. Trained using retriever aware training, Gorilla demonstrates the ability to adapt to test-time document changes, allowing flexible user updates or version changes, while substantially mitigating hallucination. To evaluate the model, the authors introduced \gls{API}Bench, a dataset consisting of HuggingFace, TorchHub, and TensorHub \glspl{API}.

To bridge the gap between the limited tool-use capabilities of open-source \glspl{LLM} and the strong tool-use performance of closed-source models such as \gls{ChatGPT}, Tool\gls{LLM} introduced a general tool-use framework encompassing data construction, model training, and evaluation~\cite{qin_toolllm_2024}. The framework centers on ToolBench, an instruction-tuning dataset constructed automatically using \gls{ChatGPT} from 16,464 real-world \gls{REST}ful \glspl{API} spanning 49 categories collected from Rapid\gls{API} Hub. \Gls{REST}ful \glspl{API} follow an architectural style, in which clients interact with server resources through \gls{HTTP} methods such as GET, POST, PUT, and DELETE, receiving responses in structured formats such as the \gls{JSON}~\cite{fielding_architectural_2000}. \Gls{HTTP} is the standard protocol used for communication between clients and servers on the web, defining how requests and responses are formatted and transmitted~\cite{berners-lee_information_1990}. \Gls{JSON} is a lightweight text-based data interchange format that represents data as human-readable key-value pairs~\cite{crockford_applicationjson_2006}. The construction process involved three stages: \gls{API} collection, instruction generation, and solution path annotation using a depth-first search-based decision tree algorithm to enhance reasoning capabilities. The resulting model, Tool\gls{LLaMA}, demonstrated the ability to execute complex instructions and generalize to unseen \glspl{API}, exhibiting performance comparable to \gls{ChatGPT}. Tool\gls{LLM} also introduced ToolEval, an automatic evaluator for assessing tool-use capabilities.

Additionally, \gls{BFCL} provides a benchmark for evaluating \glspl{LLM}’ function-calling and tool-use capabilities across real-world scenarios~\cite{patil_berkeley_2025}. \Gls{BFCL} assesses both serial and parallel function calls, supports multiple programming languages, and uses an \gls{AST}–based evaluation method that scales to thousands of functions. An abstract syntax tree is a hierarchical, tree-based representation of a program’s syntactic structure in which each node denotes a language construct and non-essential syntactic elements are omitted~\cite{perry_foundations_1992}. The benchmark combines expert-curated and user-contributed functions with associated prompts, and additionally measures models’ ability to abstain from unnecessary calls and reason in stateful, multi-step agentic interactions. Evaluations across a wide range of \glspl{LLM} show that while \gls{SOTA} models perform well on single-turn function calls, memory management, dynamic decision-making, and long-horizon reasoning remain challenging.

In terms of developing tool theory, a recent survey formally defined the function calling paradigm as the mechanism through which \glspl{LLM} ``analyze user inquiries and engage with defined functions, thereby facilitating precise responses through connections to external sources, including databases, programming interfaces, and live data streams'' ~\cite{wang_function_2026}. Unlike Toolformer, which required fine-tuning the model on filtered \gls{API} call examples, function calling provides the model with structured function schemas at inference time, typically in \gls{JSON} format, describing the function name, its purpose, and the expected parameters with their types. The model then determines whether a function is relevant to the user's query and, if so, outputs a structured object containing the function name and extracted arguments rather than a natural language response. The host application executes the function against the external system and returns the result to the model, which uses it to compose its final response. The survey describes this process as comprising three essential phases: preparation, which encompasses query analysis and function identification; execution, which evaluates whether a function call is necessary, extracts relevant parameters, and oversees the operation; and processing, which concentrates on analyzing outcomes and crafting appropriate responses.

While function calling standardized how individual \glspl{LLM} interact with external tools, it left open the question of how to standardize the communication layer between agents and the servers that host these tools, which is the problem addressed by the \gls{MCP}.

\subsubsection{Model Context Protocol}

In late 2024, Anthropic launched the \gls{MCP}, a universal open standard for defining, discovering, and invoking external tools for \gls{AI} applications~\cite{hou_model_2026}. Drawing inspiration from the Language Server Protocol, an open standard originally developed by Microsoft that decouples programming language support from code editors by defining a common interface through which any editor can communicate with any language server for features such as auto-completion and error checking~\cite{gunasinghe_language_2021}, \gls{MCP} introduces several innovations that extend beyond conventional function calling. It provides a protocol-based standard that decouples tool implementation from usage, enabling developers to publish and describe external functions dynamically, independent of any single model or agent framework. It supports dynamic discovery and schema negotiation, allowing clients to list available tools at runtime, retrieve their capabilities, and invoke them in a uniform manner without requiring prior hardcoding or platform-specific adapters. It enables bi-directional communication channels, allowing not only model-to-tool requests but also tool-initiated events and notifications back to the host. It also designs access control and capability negotiation as first-class features, offering a foundation for more auditable and secure \gls{AI}-to-tool interactions.

An example of domain-specific \gls{MCP} adoption in scientific research is the Isabl \gls{AI} Agent, developed for the Isabl Platform, a precision oncology system broadly adopted at Memorial Sloan Kettering Cancer Center and external research institutions~\cite{arango_ossa_abstract_2026}. The platform manages hundreds of projects encompassing over 105,000 sequencing experiments from approximately 70,000 individuals and more than 4.5 petabytes of analytical data. To reduce the learning curve imposed by the platform's depth and schema complexity, the authors developed an \gls{MCP} server exposing Isabl tools such as \gls{API} querying and pipeline execution through a standardized interface, combined with \gls{RAG} over indexed documentation and a ReAct-style agentic controller. ReAct is an approach that enables large language models to interleave reasoning traces and task-specific actions, allowing them to plan, update, and execute tasks while interacting with external sources for improved performance, interpretability, and reliability~\cite{yao_react_2022}. The agent handles tasks ranging from cohort discovery and multi-step genomic reasoning to automated workflow submission via natural language, demonstrating how \gls{MCP} can bridge general-purpose \glspl{LLM} with specialized scientific infrastructure.

A systematic study of \gls{MCP} defines the full lifecycle of an \gls{MCP} server as comprising four phases~\cite{hou_model_2026}. The first phase is creation, in which developers define metadata, declare capabilities, and implement the server. The second phase is deployment, in which the server is released to a platform and registered with a host. The third phase is operation, in which the server actively interprets user intent, accesses external resources, invokes tools, and manages sessions. The fourth phase is maintenance, which covers version control, configuration changes, and security auditing~\cite{hou_model_2026}. Since its release, \gls{MCP} has rapidly grown into a foundational architecture for \gls{AI}-native applications, with thousands of independently developed servers exposing model-accessible interfaces to services such as GitHub, Slack, a workplace messaging platform for team communication and collaboration~\cite{slack_technologies_llc_a_salesforce_company_slack_2026}, and Blender, an open-source three-dimensional modeling and animation application~\cite{blender_foundation_blender_2026}. Platforms like Cursor, an \gls{AI}-augmented code editor~\cite{anysphere_inc_cursor_2026}, and Claude Desktop, a desktop application providing access to Anthropic's \glspl{LLM}~\cite{anthropic_pbc_use_2026}, demonstrate how \gls{MCP}-enabled clients can flexibly extend functionality by connecting to new servers on demand.

A second study examined the architectural role of the \gls{MCP} in multi-agent systems, interpreting it not merely as an integration protocol but as an autonomous architectural layer that formalizes the topology of interaction among hosts, clients, and servers~\cite{zuykov_architectural_2026}. The study identified four mutually reinforcing principles derived from the protocol's design. The first is strict separation of concerns, which keeps cognitive functions such as planning and synthesis distinct from execution and data access. The second is security and governance by design, which relocates access control and auditability to the server layer. The third is modular composability and interoperability, which enables shared capability providers to serve multiple agents across heterogeneous frameworks. The fourth is declarative context provisioning, which transforms prompts and resources into managed assets that can be discovered, versioned, and reused across workflows. The study also proposed a lifecycle interpretation showing that \gls{MCP} stabilizes system evolution across creation, operation, and update phases, with a customer support reference architecture demonstrating how narrowly scoped cognitive roles can be coordinated by a host while relying on a shared capability layer. However, the study noted that the broader ecosystem still faces unresolved challenges around centralized oversight, multi-tenant configuration, cross-component authentication, and trusted distribution.

Apart from \glspl{API} and the \gls{MCP}, which standardize how agents discover and invoke remote tools, the \gls{CLI} \gls{BASH} has become an increasingly popular type of tool for software and coding agents that need to interact directly with the local operating system and any \gls{CLI}-based tool installed on the host machine.

\subsubsection{Command Line Interface}

A \gls{CLI} is a text-based interface through which users or programs issue commands as typed strings and receive text output in return. The concept can be traced at least as far back as early time-sharing systems, which allow multiple users to interact with a single computer concurrently by rapidly switching between their tasks. One example is the Compatible Time-Sharing System developed at \gls{MIT} in the early 1960s, which provided users with an interactive command interpreter for managing files and executing programs from a terminal~\cite{corbato_experimental_1962}. The established synonym \gls{CLI}, emphasizing the user-facing aspect rather than the underlying interpreter, appeared as early as 1983 in research on user-defined interfaces for electronic mail systems~\cite{wixon_building_1983}.

Unix, a general-purpose, multi-user, interactive operating system developed in the early 1970s, played a pivotal role in shaping the modern \gls{CLI}~\cite{ritchie_unix_1974}. It introduced foundational abstractions such as a hierarchical file system, uniform treatment of files and devices through a common input/output interface, and process control primitives that remain central to modern operating systems. The primary means of user interaction with Unix was the Shell, a \gls{CLI} that reads lines of text typed by the user and interprets them as requests to execute programs. The Shell runs as an ordinary user-level process rather than as a privileged part of the operating system kernel, a design decision that established the enduring separation between the operating system and its command interface.

The Bourne Shell, introduced by Stephen Bourne as a successor to the original Unix shell, unified interactive command execution with a general-purpose programming language, establishing the shell not merely as a \gls{CLI} but as a scripting environment~\cite{bourne_introduction_1997}. The Bourne Shell reads and executes commands either interactively from a terminal or from stored files known as shell procedures. Core features include input and output redirection, which allows commands to read from and write to files rather than the terminal. Pipelines connect the standard output of one command to the standard input of another using the pipe operator. File name generation through pattern matching with wildcard characters and control-flow primitives such as \texttt{for}, \texttt{while}, \texttt{if}, and \texttt{case} enable the construction of complex command procedures.

The behavior of the Bourne Shell was later formalized by the \gls{IEEE} as part of the \gls{POSIX} Shell and Utilities specification, a family of open system standards based on Unix that defines the command language, built-in utilities, and interactive facilities a conformant shell must provide~\cite{ramey_whats_1994}. \Gls{BASH} is an implementation of this standard that extends the original Bourne Shell with additional features such as command-line editing, command history, and job control~\cite{ramsey_gnu_2025}. It can be used interactively, accepting input typed from a keyboard, or non-interactively, executing commands read from a file or a string. It supports both synchronous execution, where the shell waits for a command to complete before accepting further input, and asynchronous execution, where commands run in parallel with the shell.

An early exploration of \gls{AI} agents operating \glspl{CLI} was Project \Gls{CLAI}~\cite{agarwal_project_2020}. The project framed the \gls{CLI} as a generic environment that researchers could interface with using a simple sense-act \gls{API}, following the traditional agent architecture. \Gls{CLAI} provided a heavily instrumented version of \gls{BASH}, enabling new end-user interaction patterns such as natural language translation of command-line tasks to their correct \gls{BASH} syntax.

It was open-sourced at the fourth \gls{AAAI} Conference on Artificial Intelligence and led to the \Gls{NLC2CMD} competition at the \gls{NeurIPS} 2020, which challenged participants to translate natural language descriptions of command-line tasks into valid \gls{BASH} commands~\cite{agarwal_neurips_2021}. The competition revealed several open challenges, including the difficulty of handling underdetermined invocations where natural language descriptions map to multiple valid command translations, the need for session-level evaluation rather than isolated single-command accuracy, and security concerns around command injection through manipulated training data. Participants also noted that the frequency of real-world commands follows a heavily tailed distribution, suggesting that focusing on a restricted but frequently used set of commands may yield more practical accuracy than attempting full coverage.

Over the following years, work has focused on improving the evaluation of natural language to \gls{BASH} command translation. A study presented a manually verified test dataset of 600 instruction-command pairs and a training dataset of 40,939 pairs, increasing the size of previous datasets by 441\% and 135\% respectively~\cite{westenfelder_llm-supported_2025}. The study also introduced a novel functional equivalence heuristic that combines command execution with \gls{LLM} evaluation of command outputs, determining the functional equivalence of two \gls{BASH} commands with 95\% confidence, a 16\% increase over previous heuristics. Evaluation of popular \glspl{LLM} demonstrated that parsing, in-context learning, in-weight learning, and constrained decoding can improve translation accuracy by up to 32\%. However, the accuracy of \gls{SOTA} \glspl{LLM} for this task remains low, highlighting that natural language to \gls{BASH} translation continues to be a challenging problem.

\subsection{Skills}

The concept of skills in multi-agent systems emerged in robotics research in the late 20th century. An early formalization defined skills as the learned ability to perform competent actions toward a specified goal, using available sensor data and a system's actuation capabilities~\cite{friedrich_integrating_1998}. In this context, a skill produces a sequence of actions that change the world state over time toward a desired goal state, with termination and error criteria embedded to recognize successful execution or failure. A key challenge identified was that skills are inherently agent-specific, meaning that even skills producing equivalent effects are represented and implemented differently across different agents. To address this, the concept of elementary operations was proposed as an interfacing structure that encapsulates agent-specific skill implementations within an agent-independent symbolic representation, supporting task negotiation, planning, and inter-agent communication within the multi-agent system. This separation between a skill's external description and its internal implementation anticipated a pattern that would later re-emerge in the design of \gls{LLM}-based \gls{AI} agents.

The concept of skill has also been applied to the evaluation of agent performance in multi-agent systems. A study on competitive social environments introduced the Relative Skill-Level Estimator, a method for rating the skill level of synthetic agents as an alternative to established human-oriented rating systems such as Elo and Glicko~\cite{kiourt_rating_2019}. The Elo rating system, originally developed for chess, assigns each player a numerical rating that increases or decreases based on the outcome of games against other rated players, with the magnitude of change depending on the rating difference between opponents~\cite{elo_rating_1978}. The Glicko system extends Elo by incorporating a reliability measure that accounts for how frequently a player competes, assigning less certainty to ratings of inactive players~\cite{glickman_glicko_2016}. The authors found that these methods, originally designed for human competition, showed significant inconsistency when applied to large-scale multi-agent settings where matches between agents of very different ability levels occur frequently. In contrast, the proposed method demonstrated more stable and accurate ratings while being simpler to compute. These findings suggested that as multi-agent systems grow in scale and complexity, skill evaluation methods designed specifically for agent populations are needed rather than direct adaptations of approaches developed for human competition.

In the context of \glspl{LLM}-based \gls{AI} agents, skills are defined as ``abstractions over useful and reusable behaviors for the target tasks''~\cite{liu_skillact_2024}. SkillAct proposed a prompting approach in which each skill is represented as a tuple of a name, a set of instructions describing how to execute the skill, and examples of the skill in use extracted from task trajectories. By appending these skill descriptions to an \gls{LLM}-based agent's prompt, the agent gains access to reusable behavioral templates that guide its planning and action execution across different tasks. The study demonstrated that skills can also be automatically extracted from few-shot demonstrations by prompting an \gls{LLM} to identify shared behaviors across trajectories, producing skills comparable in quality to those written by human experts. Evaluated on ALFWorld, a text-based simulated household environment built on top of the \gls{ALFRED} benchmark for embodied agents completing household tasks in visually grounded settings~\cite{shridhar_alfworld_2021}, SkillAct improved success rates from 79.7\% to 85.1\% with extracted skills and to 88.0\% with expert-written skills when using \gls{GPT}-4-turbo, with analysis showing that skills primarily reduced plan execution errors such as incorrect action syntax rather than planning errors. This notion of skills as reusable, composable behavioral abstractions encoded in natural language has since been adopted by practical coding agent tools, where skills take the form of \gls{MD} files containing instructions that extend the agent's capabilities and can be invoked automatically or on demand. A \gls{MD} file is a plain text file formatted using \gls{MD} syntax, which allows structured text, headings, code blocks, and lists to be easily written, and read~\cite{gruber_daring_2004}.

Recently, these abstractions have been implemented as agentic coding manifests, ``configuration files (such as Claude.\gls{MD}) that provide agents with essential project context, identity, and operational rules''~\cite{scanniello_use_2026}. A Claude.\gls{MD} file is a markdown file placed in a project's root directory that is automatically loaded into the context of Claude Code, Anthropic's coding agent~\cite{anthropic_pbc_claude_2026}, at the start of every session, serving as persistent instructions that shape how the agent understands and interacts with the codebase~\cite{anthropic_pbc_how_2026}. More recently, the AGENTS.\gls{MD} format has emerged as an open, standardized alternative, making these practices broadly compatible across different \gls{AI} coding agents~\cite{lf_projects_llc_agentsmd_2026}.

A study analyzing 253 such files from 242 repositories found that developers prefer a shallow hierarchical structure, typically using a single primary heading with a moderate number of subsections~\cite{scanniello_use_2026}. Content is dominated by operational commands, technical implementation notes, and high-level architecture descriptions, with many manifests also including \gls{AI} role definitions to guide agent behavior. These findings highlight the dual purpose manifests serve in agentic coding workflows: they function both as execution guides for technical tasks and as frameworks for human-\gls{AI} collaboration. This pattern echoes the elementary operations concept from robotics, where a single structure encapsulated both the agent-independent description needed for coordination and the agent-specific implementation needed for execution.

While the preceding chapters have established the theoretical foundations of agents, tools, and skills, the following chapter examines how these concepts have been applied and evaluated in existing research.

\section{Related Work}
\glsresetall

This section reviews existing research relevant to the present work. It is organized into three subsections: \gls{ML}, benchmarks, and tool design and optimization. The first subsection examines \gls{ML} from two perspectives: agents that automate \gls{ML} workflows and the platforms on which these workflows are executed. The second covers benchmarks across four areas: coding agents, the \gls{MCP}, the \gls{CLI}, and \gls{ML} agents. The third addresses the design and optimization of tools available to agents.

\subsection{Machine Learning}

This subsection reviews the intersection of agents and platforms in the context of \gls{ML}. It first examines existing approaches to specializing software agents for automating \gls{ML} workflows, analyzing their pipeline coverage, tool interface design, and degree of platform integration. It then surveys the broader landscape of \gls{ML} platforms, characterizing their support for \glspl{CLI}, the \gls{MCP}, agents, and skills.

\subsubsection{Agents}

Various approaches have been proposed to create specialized agents for automating ML workflows, giving rise to a class of systems commonly referred to as \gls{ML} agents. These agents leverage \glspl{LLM} to reduce manual effort across stages of the \gls{AutoML} pipeline and vary in their degree of platform integration, tool interface design, and pipeline coverage. Existing work can be grouped into three categories: agents that address the complete \gls{AutoML} pipeline without platform or tool integration, agents that employ tools but are limited to a single interface type or pipeline stage, and agents with platform integration but without standardized tool interfaces.

One agent has been proposed that enables natural language interaction throughout the entire \gls{ML} workflow~\cite{huang_human-centered_2025}. The agent implements an end-to-end \gls{ML} pipeline incorporating automatic data loading and pre-processing, task identification, neural architecture selection, hyperparameter optimization, and training automation. It introduces a data processing approach that leverages \glspl{LLM} to automatically interpret and handle diverse data formats without requiring manual pre-processing, as well as an adaptive hyperparameter optimization strategy that combines \gls{LLM} knowledge of \gls{ML} best practices with dynamic performance feedback. Evaluation on ten datasets spanning classification and regression tasks demonstrates increased performance compared to traditional rule-based \gls{AutoML} frameworks. While the agent covers the complete \gls{AutoML} pipeline including data preparation, feature engineering, model generation, and model evaluation, it operates through remote data ingestion rather than integration with an \gls{ML} platform.

\gls{I-MCTS} addresses the low-diversity and suboptimal code generation that agents in \gls{AutoML} often exhibit by introducing an introspective process that iteratively expands tree nodes through analysis of solutions and results from parent and sibling nodes~\cite{liang_i-mcts_2026}. The approach integrates an \gls{LLM}-based value model to evaluate each node's solution prior to computational roll-outs, and implements a hybrid rewarding mechanism that transitions from \gls{LLM}-estimated scores to actual performance scores. Applied to various \gls{ML} tasks, \gls{I-MCTS} demonstrates a 4\% absolute improvement over strong open-source \gls{AutoML} agents. While the approach targets the complete \gls{AutoML} pipeline, it focuses solely on the agent's search and reasoning strategy without considering platform integration or tool interfaces such as the \gls{MCP} or the \gls{CLI}.

Within data lake systems, a second line of work integrates \gls{AutoML} frameworks and \glspl{LLM} into the \gls{SEDAR} platform~\cite{hoseini_towards_2025}. \gls{SEDAR} is a semantic data lake that integrates data ingestion, storage, processing, governance, and semantic enrichment to support data integration, linking, profiling, and \gls{ML} workflows~\cite{hoseini_sedar_2023}.  The work introduces a metadata model for capturing data analytics processes, a Python package wrapping existing \gls{AutoML} libraries, and a module utilizing \glspl{LLM} to automate \gls{ML} tasks. A comparative evaluation indicates that \gls{AutoML} simplifies pipeline creation but limits user control and lacks robust data pre-processing support, while \glspl{LLM} can automate individual tasks such as code generation but struggle to orchestrate complete workflows effectively. The authors propose an extended vision integrating multi-agent frameworks with knowledge graphs that capture historical experience from previous \gls{ML} experiments. This approach addresses data preparation, model generation, and model evaluation, but does not employ the \gls{MCP} or the \gls{CLI} as tool interfaces, instead relying on an integrated agent architecture within the data lake system.

A third study transforms a conventional \gls{API}-based analytics platform into a set of \gls{MCP}-based tools accessible by agents, establishing a process to facilitate communication between data analysts, agents, and the platform~\cite{fikardos_generative_2026}. The resulting chat-based interface allows analysts to query and execute analytical workflows using natural language. The framework is instantiated with open-source models and achieves a 7.2\% mean overall score increase compared to baselines, with complementary user study data demonstrating superior task efficiency and higher user preference scores relative to a traditional form-based interface. While this approach employs the \gls{MCP} as its tool interface, it is limited to the data preparation stage of the \gls{AutoML} pipeline and does not address feature engineering, model generation, or model evaluation.

Auto\gls{ML}Pipeline is a closed-loop code generation framework that integrates \gls{RAG} with \gls{RL} feedback mechanisms to produce deployment-ready pipeline code for cloud-native \gls{ML} platforms~\cite{zhao_automl-pipeline_2026}. The framework leverages a knowledge base constructed from successful pipeline execution logs to guide \gls{GPT}-4 in generating code that adheres to platform-specific constraints, and introduces a pre-validation agent that employs simulated execution environments to predict resource consumption and detect dependency conflicts before deployment. Evaluation on the CodeSearchNet dataset augmented with Azure \gls{ML} pipeline specifications demonstrates a 43.7\% improvement in first-submission success rate and a 31.2\% reduction in resource over-provisioning compared to vanilla \gls{GPT}-4 baselines. This approach is the closest to the present work in that it generates executable code targeting a cloud platform, covering data preparation, model generation, and model evaluation. However, Azure is a general-purpose cloud platform rather than a special-purpose \gls{ML} platform, the framework does not address feature engineering, and it relies solely on code generation without employing external tools such as the \gls{MCP} or the \gls{CLI} to interact with the platform at runtime.

In summary, while existing research has explored agents for automating \gls{ML} workflows, no study to date has compared the performance of agents across different tool types when operating \gls{ML} platforms entirely via their interfaces. Current approaches either rely on integrated architectures without standardized tool interfaces, limit their scope to individual stages of the \gls{AutoML} pipeline, or generate code without runtime interaction with the target platform. None of these works evaluate how the choice of tool type, such as the \gls{MCP} or the \gls{CLI}, affects agent performance in managing feature, training, and inference pipelines on a \gls{ML} platform, nor do they identify the conditions under which one tool type is more suitable than the other. Furthermore, while skills have been shown to improve agent performance, whether this effect holds equally across tool types when operating \gls{ML} platforms remains unexplored.

\subsubsection{Platforms}

Beyond the Hopsworks Feature Store, numerous other data science and \gls{ML} platforms exist. Leaders have mature platform strategies that leverage \gls{GenAI} and agents, and innovate at a pace that outpaces competing vendors. Eight platforms occupy this category, including those from Databricks, Google Cloud, Microsoft Azure and \gls{AWS}. Challengers possess the operational capacity to serve broad enterprise needs but are constrained by limited geographic or industry focus. Alibaba Cloud is the sole entrant in this category. Visionaries offer differentiated, forward-looking solutions but lack sufficient recognition for end-to-end adoption, and include five platforms from companies such as Snowflake, \gls{SAS} and Cloudera. Niche Players, such as Alteryx and MathWorks, focus on specific industries or user groups but have limited broader market traction~\cite{wiley_databricks_2025, jaffri_magic_2025}.

Databricks recently released Genie Code, a coding agent that can build pipelines, debug failures, train \gls{ML} models, and proactively monitor production workloads directly within the platform~\cite{wendell_introducing_2026}.

Despite the increasing adoption of \gls{ML} agents and \gls{MCP} in \gls{ML} platforms, no research exists that systematically evaluates whether these mechanisms enable effective end-to-end usage of such platforms, or how performance differs when agents interact with these platforms via \gls{CLI} or \gls{MCP} interfaces. While some empirical evidence suggests that skills can improve agent performance, these findings are not derived from \gls{MLOps} platform settings and do not assess their impact on agents’ ability to operate \gls{ML} platforms across the entire \gls{AutoML} process. Having outlined the current landscape of both agents platforms and their limitations in the context of \gls{ML}, the following section examines how these systems are evaluated.

\subsection{Benchmarks}

The following subsections review existing benchmarks across four areas relevant to the \glspl{RQ}: coding agents, the \gls{MCP}, the \gls{CLI}, and \gls{ML} agents.

\subsubsection{Coding Agents}

Existing benchmarks for coding agents span a broad range of capabilities, including safety, data science, autonomous task execution, repository-level code generation, automated program repair, communication skills, and collaborative coding.

In the domain of safety, RedCode provides an evaluation platform for assessing the security of code agents across unsafe code execution and generation~\cite{guo_redcode_2024}. It covers 25 types of critical vulnerabilities spanning websites, file systems, and operating systems, and evaluates agents across multiple input formats including Python, Bash, and natural language descriptions. Findings indicate that coding agents are more likely to reject unsafe operating system operations than technically buggy code, and that more capable base models tend to produce more sophisticated harmful software.

Several benchmarks target code generation in data-intensive and repository-level contexts. \gls{DA}-Code evaluates \glspl{LLM} on agent-based data science tasks grounded in real and diverse data, requiring complex data wrangling and analytics using Python and the \gls{SQL}~\cite{huang_da-code_2024}. AppWorld provides an execution environment of nine applications operable via 457 \glspl{API}, with 750 autonomous agent tasks requiring rich and interactive code generation~\cite{trivedi_appworld_2024}. CodeAgent introduces a framework with five programming tools for repository-level code generation, evaluated on the CodeAgentBench dataset and HumanEval~\cite{zhang_codeagent_2024}. GitTaskBench assesses coding agents on 54 realistic tasks across seven modalities and seven domains, pairing repositories with automated evaluation harnesses and introducing an alpha-value metric to quantify economic benefit~\cite{ni_gittaskbench_2026}. ProjectEval simulates user interaction for project-level code generation evaluation across three input levels~\cite{liu_projecteval_2025}.

Automated program repair represents another active area. A survey traces the evolution from template and constraint-based approaches to \gls{LLM}-powered systems, comparing retrieval-augmented, agent-based, and agentless paradigms across benchmarks such as \gls{SWE}-bench, CodeAgent-Bench, and Code\gls{RAG}-Bench~\cite{puvvadi_coding_2025}. \gls{MDC}Agent addresses model diagram-to-code generation through a collaborative multi-agent framework for parsing, generating, and verifying code from model diagrams across 16 research domains~\cite{wang_model_2025}.

Further benchmarks evaluate broader coding agent capabilities. \gls{MMAU} provides offline evaluation across five domains including tool use, data science and \gls{ML} coding, and contest-level programming, covering capabilities such as understanding, reasoning, planning, problem-solving, and self-correction~\cite{yin_mmau_2025}. HumanEvalComm assesses the communication skills of \glspl{LLM} for code generation, evaluating their ability to ask clarifying questions when problem descriptions contain inconsistencies, ambiguities, or incompleteness~\cite{wu_humanevalcomm_2025}. CooperBench evaluates collaborative coding through 600 tasks where two coding agents must coordinate on independently implementable but potentially conflicting features, revealing a 30\% average drop in success when agents work together versus individually~\cite{khatua_cooperbench_2026}. \gls{CoSQA}+ introduces an automated pipeline for semantic code search evaluation with test-driven agent annotation~\cite{gong_cosqa_2026}. BenchAgents proposes a framework that leverages \glspl{LLM} to automate benchmark creation for complex capabilities through planning, generation, data verification, and evaluation agents~\cite{butt_benchagents_2025}. 

While \gls{MMAU} includes data science and \gls{ML} coding as one of its evaluation domains, it assesses general coding competence on \gls{ML} tasks rather than evaluating coding agents that operate \gls{ML} platforms through tool interfaces. None of the reviewed benchmarks evaluate coding agents managing feature, training, and inference pipelines on a \gls{ML} platform, nor do they compare agent performance across different tool types such as the \gls{MCP} and the \gls{CLI}. Furthermore, existing coding agent benchmarks do not consider the influence of skills written in \gls{MD} on coding agent performance.

\subsubsection{Model Context Protocol}

Several benchmarks have been proposed to evaluate \gls{LLM} agent performance within the \gls{MCP} framework, spanning general-purpose tool interaction, domain-specific applications, and security.

The first benchmark addresses the application of the \gls{MCP} to relational database exploration~\cite{bramer_toolbelt-mcp_2026}. The system enables an \gls{LLM} to respond to user queries by generating and executing query-specific \gls{SQL}, supported by a tool for discovering relational paths between tables through a graph of transitively connected foreign-key relationships. While this approach demonstrates the viability of the \gls{MCP} for dynamic data access, its scope is limited to data retrieval and schema navigation within relational databases. It does not extend to feature engineering, model generation, or model evaluation, restricting its relevance to the data preparation stage of the \gls{AutoML} pipeline.

\gls{MCP}-AgentBench provides a comprehensive benchmark for assessing language agent capabilities in \gls{MCP}-mediated tool interactions~\cite{guo_mcp-agentbench_2026}. It establishes a testbed comprising 33 operational servers with 188 distinct tools and includes 600 systematically designed queries distributed across six categories of varying interaction complexity. Evaluation is conducted using \gls{MCP}-Eval, an outcome-oriented methodology prioritizing real-world task success. Similarly, \gls{MCP}Eval is an open-source \gls{MCP}-based framework that automates end-to-end task generation and evaluation of \gls{LLM} agents across diverse domains~\cite{liu_mcpeval_2025}. It standardizes metrics, integrates with native agent tools, and eliminates manual effort in building evaluation pipelines. Empirical results across five real-world domains demonstrate its effectiveness in revealing domain-specific performance differences. Both benchmarks target general-purpose agent-tool interaction and do not incorporate tasks related to data preparation, feature engineering, model generation, or model evaluation.

\gls{MCP}-Bench evaluates \gls{LLM} agents on realistic, multi-step tasks requiring tool use, cross-tool coordination, and planning across 28 live \gls{MCP} servers spanning 250 tools in domains such as finance, travel, scientific computing, and academic search~\cite{wang_mcp-bench_2026}. Among its servers, it includes an \gls{AI} model hub integration service that provides access to \gls{ML} models, datasets, and research papers through tools for searching and retrieving model and dataset information. This server partially addresses data preparation and model evaluation by enabling dataset discovery and model comparison. However, it is limited to retrieval of existing resources and does not support feature engineering or model generation, such as training models or constructing \gls{AutoML} pipelines.

In the domain of security, \gls{MCP}Tox introduces the first benchmark to systematically evaluate agent robustness against tool poisoning in realistic \gls{MCP} settings~\cite{wang_mcptox_2026}. Constructed upon 45 live, real-world \gls{MCP} servers and 353 authentic tools, it generates 1,348 malicious test cases covering ten categories of potential risks. Evaluation on 20 prominent \gls{LLM} agents reveals widespread vulnerability, with more capable models often more susceptible due to their superior instruction-following abilities. While \gls{MCP}Tox addresses an important dimension of agent evaluation, it focuses on security rather than task performance across \gls{ML} workflows.

However, none of the reviewed \gls{MCP} benchmarks evaluate agents managing feature, training, and inference pipelines on a \gls{ML} platform, nor do they compare \gls{MCP}-based tool interaction against alternative tool types such as the \gls{CLI}. Furthermore, none of the existing \gls{MCP} benchmarks incorporate the effect of skills on agent performance.

\subsubsection{Command Line Interface}

Compared to the \gls{MCP}, benchmarking of \gls{LLM} agents operating \glspl{CLI} remains limited. Terminal-Bench 2.0 is the only benchmark specifically designed to evaluate agents on tasks in computer terminal environments~\cite{merrill_terminal-bench_2026}. It comprises 89 curated tasks inspired by real workflows, each featuring a unique environment, a human-written solution, and comprehensive tests for verification. Among its tasks, the benchmark includes several \gls{ML}-related challenges~\cite{stanford_university_terminal-bench_2026}. One task requires training a \gls{CNN} on \gls{CIFAR}-10, a widely used image classification dataset consisting of 60,000 colour images across ten classes~\cite{krizhevsky_learning_2009}, using the Caffe framework, an early open-source \gls{DL} library developed at the Berkeley Vision and Learning Center~\cite{jia_caffe_2014}. Another involves reconstructing and fine-tuning a PyTorch model from a state dictionary~\cite{paszke_pytorch_2019}, a serialized representation of a model's learned parameters. A third task requires implementing pipeline parallel training for the \gls{LLaMA} model, a technique that partitions model layers across multiple devices to enable training of large models that exceed the memory capacity of a single device. A fourth task involves training a FastText model, a lightweight text classification and word representation library developed by Facebook Research~\cite{joulin_fasttextzip_2016}, on review data. A fifth task requires building a command line tool for inference on \gls{MNIST}, a benchmark dataset of handwritten digit images commonly used for evaluating image classification models~\cite{lecun_mnist_2010}. These tasks partially address the model generation and model evaluation stages of the \gls{AutoML} pipeline. Evaluation of frontier models and agents shows that none exceeds 65\% on the benchmark, with error analysis identifying areas for model and agent improvement. However, Terminal-Bench 2.0 does not systematically cover the full \gls{AutoML} pipeline, including data preparation and feature engineering, nor does it evaluate agents operating a \gls{ML} platform or compare \gls{CLI}-based interaction against alternative tool types such as the \gls{MCP}. 

In summary, no \gls{CLI} benchmark for \gls{MLOps} exists, and no studies compare the performance of \gls{CLI}-based tools with \gls{MCP}-based tools in the context of \gls{ML} platforms. Furthermore, there is no systematic empirical evidence on how skills influence \gls{CLI} usage in coding agents operating \gls{ML} platforms.

While the preceding sections examined benchmarks for coding agents, the \gls{MCP}, and the \gls{CLI} in general, the following section reviews benchmarks that specifically target agents performing \gls{ML} tasks.

\subsubsection{Machine Learning Agents}

Several benchmarks have been proposed to evaluate agents specifically on \gls{ML} tasks, ranging from experimentation and engineering to code generation for \gls{ML} algorithms.

\gls{ML}AgentBench introduces a suite of 13 tasks designed to assess whether \gls{LLM}-driven agents can perform \gls{ML} experimentation effectively~\cite{huang_mlagentbench_2024}. Tasks range from improving model performance on \gls{CIFAR}-10 to recent research problems such as \gls{BabyLM}, a shared task focused on training language models on developmentally plausible amounts of data~\cite{edman_mask_2025}. Agents can perform actions such as reading and writing files, executing code, and inspecting outputs, following the ReAct framework. Evaluation across several frontier models finds that a Claude v3 Opus agent achieves the highest success rate at 37.5\%, though performance varies considerably, spanning from 100\% on well-established datasets to 0\% on recent Kaggle challenges. Kaggle is an online platform that hosts \gls{ML} competitions in which participants build predictive models on shared datasets and are ranked on a public leaderboard according to a predefined evaluation metric \cite{kaggle_inc_kaggle_2026}. The benchmark identifies long-term planning and hallucination reduction as key challenges. While \gls{ML}AgentBench evaluates agents across multiple stages of \gls{ML} experimentation, it does not assess agents operating a \gls{ML} platform through tool interfaces such as the \gls{MCP} or the \gls{CLI}.

\gls{ML}Gym introduces the first Gym environment for \gls{ML} tasks, providing both a framework and the benchmark \gls{ML}Gym-Bench for evaluating and developing \gls{LLM} agents on \gls{AI} research tasks~\cite{nathani_mlgym_2025}. \gls{ML}Gym-Bench comprises 13 diverse, open-ended research tasks spanning computer vision, natural language processing, \gls{RL}, and game theory. Solving these tasks requires a broad range of research skills, including generating hypotheses, creating and processing data, implementing \gls{ML} methods, training models, running experiments, analyzing results, and iterating to improve performance. By framing these tasks as a Gym environment, \gls{ML}Gym additionally enables research on \gls{RL} algorithms for training such agents. Evaluation of several frontier models, including Claude 3.5 Sonnet, \gls{GPT}-4o, o1-preview, Llama-3.1 405 \gls{B}, and Gemini-1.5 Pro, shows that current models can improve on given baselines, typically by finding better hyperparameters, but do not generate novel hypotheses, algorithms, or architectures. While \gls{ML}Gym broadens the scope of \gls{ML} agent evaluation to open-ended research and provides an extensible framework for developing new learning algorithms, it does not evaluate agents interacting with a \gls{ML} platform through standardised tool interfaces.

Building on \gls{ML}AgentBench, \gls{MLE}-bench curates 75 \gls{ML} engineering competitions from Kaggle to measure how well \gls{AI} agents perform at real-world \gls{ML} engineering tasks such as training models, preparing datasets, and running experiments~\cite{chan_mle-bench_2025}. Human baselines are established using Kaggle's publicly available leaderboards. The best-performing setup, \gls{OpenAI}'s o1-preview with \gls{AIDE}, achieves at least the level of a Kaggle bronze medal in 16.9\% of competitions. \Gls{AIDE} is a \gls{ML} engineering agent that frames ML engineering as a code optimization problem, using tree search over potential solutions to iteratively refine and reuse promising approaches~\cite{jiang_aide_2025}. The benchmark additionally investigates resource scaling for \gls{AI} agents and the impact of pre-training contamination. \Gls{MLE}-bench provides a broader and more challenging evaluation than \gls{ML}AgentBench, but similarly does not evaluate agents interacting with a \gls{ML} platform through standardised tool interfaces.

\gls{ML}Algo-Bench, which references \gls{MLE}-bench, introduces two tasks focused on \gls{ML} code generation~\cite{wang_mlalgo-bench_2025}. The first requires generating code for \gls{ML} algorithms including both traditional and deep learning-based methods. The second provides human-authored solution sketches and requires writing \gls{ML} code for practical tasks in Kaggle competitions. Evaluation using an automatic framework with metrics such as task pass rate, relative performance, and time overhead shows that the top-performing model at the time of publication, Claude 3.5 Sonnet, achieves a 48.8\% completion rate on algorithm implementation and 21.6\% on Kaggle competitions. While \gls{ML}Algo-Bench evaluates model generation and model evaluation, it does not address data preparation or feature engineering as autonomous stages, nor does it assess agents operating \gls{ML} platforms through tool interfaces.

In summary, existing \gls{ML} agent benchmarks evaluate agents on experimentation, engineering, and code generation tasks, but none assesses agents managing feature, training, and inference pipelines on a \gls{ML} platform. Furthermore, no benchmark compares agent performance across different tool types such as the \gls{MCP} and the \gls{CLI}, identifies the conditions under which each tool type is more suitable, or examines the role of skills in shaping agent behavior across these tool types.

\subsubsection{Tool Design and Optimization}

Existing research on tool design for \gls{LLM} agents focuses primarily on optimizing tool selection and context management as the number of available tools scales beyond the constraints of the context window.

A first study delivers the first large-scale mixed-methods analysis of strategies that reconcile multi-\gls{MCP} autonomy with context limits~\cite{v_scaling_2025}. It benchmarks six optimization families, including baseline, \gls{RAG}-\gls{MCP}, contextual compression, hierarchical \gls{MCP} routing, adaptive chunking, and multi-agent coordination, across quantitative metrics and eight qualitative case studies. Experiments on TaskBench, \gls{MCP}-Radar, \gls{MCP}Eval, and \gls{BFCL} show a mean token reduction of 37\% and a mean accuracy gain of 23 percentage points. The study synthesizes a decision matrix and proposes a governance framework for enterprise deployments, positioning contextual compression combined with \gls{RAG}-\gls{MCP} as the current Pareto frontier for developers.

MemTool introduces a short-term memory framework enabling \gls{LLM} agents to dynamically retrieve and manage tools or \gls{MCP} server contexts across multi-turn conversations~\cite{campos_memtool_2026}. It offers three agentic architectures: an autonomous agent mode granting full tool management autonomy, a workflow mode providing deterministic control, and a hybrid mode combining both. Evaluation across 12 \glspl{LLM} on Scale\gls{MCP}-334K, a benchmark of 334,685 tools, over 100 consecutive user interactions shows that reasoning \glspl{LLM} achieve high tool-removal efficiency of 90--94\% in autonomous mode, while medium-sized models exhibit significantly lower efficiency. Workflow and hybrid modes consistently manage tool removal effectively, whereas autonomous and hybrid modes excel at task completion.

ScaleMCP proposes a tool selection approach that dynamically equips \gls{LLM} agents with an \gls{MCP} tool retriever and an auto-synchronizing tool storage pipeline through CRUD operations with \gls{MCP} servers as the single source of truth~\cite{v_scaling_2025}. It introduces Tool Document Weighted Average, an embedding strategy that selectively emphasizes critical components of tool documents during the embedding process. Evaluation on a dataset of 5,000 financial metric \gls{MCP} servers across ten \glspl{LLM}, five embedding models, and five retriever types demonstrates substantial improvements in tool retrieval and agent invocation performance.

While these studies advance the optimization of tool selection and context management for \gls{LLM}-based agents, none compare agent performance across different tool types such as the \gls{MCP} and the \gls{CLI} or identify the conditions under which each is more suitable. Furthermore, no existing research addresses how \gls{CLI} tools should be constructed to be used effectively by coding agents operating \gls{ML} platforms, nor whether the effect of skills on agent performance differs between tool types.

\subsection{Summary}
\glsresetall

This chapter presented the foundational concepts and related work underpinning this thesis. The progression began with \gls{AI}, broadly defined as algorithms capable of performing tasks that typically require human intelligence, and narrowed to \gls{ML}, its subfield concerned with algorithms that learn to solve tasks from data without explicit programming. \Gls{ML} engineering introduced the practical principles governing effective model development, including the interplay of representation, evaluation, and optimization, the challenge of overfitting, and the structured \gls{AI} lifecycle spanning design, development, and deployment. \Gls{MLOps} then extended these principles to the operational level, drawing on DevOps practices such as \gls{CI/CD} to enable fast, reproducible, and reliable releases of \gls{ML} applications. \Gls{ML} platforms were presented as the infrastructure supporting these practices, with the Hopsworks Feature Store illustrating how feature pipelines, training pipelines, and inference pipelines can be organized around a centralized feature store with support for batch, streaming, and request-time computation.
The chapter then examined \gls{DL}, the subset of \gls{ML} that learns hierarchical representations through multiple successive layers, reducing the need for manual feature engineering. Key architectures discussed included \glspl{RvNN} for hierarchical data, \glspl{RNN} and \glspl{LSTM} for sequential data, and \glspl{CNN} for spatially structured data. \Gls{GenAI} was introduced as the branch of \gls{DL} concerned with generating previously unseen synthetic content, tracing the evolution from early probabilistic models such as \glspl{RBM} and \glspl{DBN} through \glspl{VAE} and \glspl{GAN} to the transformer architecture, which replaced recurrent mechanisms with self-attention to enable parallelized training. \Glspl{LLM} were then presented as the class of transformer-based models capable of executing diverse \gls{NLP} operations, with key concepts including tokenization, attention mechanisms, \gls{RL} from human feedback, in-context learning, the constraints imposed by the context window, and the biases these models encode and propagate from their training data. Code generation models were discussed as \glspl{LLM} adapted or purpose-built for \gls{SWE} tasks, evaluated through benchmarks such as \gls{SWE}-bench, and situated within a broader ecosystem of model hubs, licensing frameworks, and reuse strategies. Vibe coding was introduced as the emerging development practice that leverages these models, characterized by conversational code generation, multimodal expression processing, and context-aware generation patterns. \Gls{AutoML} was then discussed as the automated construction of \gls{ML} pipelines on a limited computational budget, comprising data preparation, feature engineering, model generation, and model evaluation, with \gls{LLM} contributions reviewed across all four stages.

The discussion then shifted to agents, beginning with intelligent agents defined by the properties of autonomy, social ability, reactivity, and proactiveness, and formalized through the \gls{BDI} model of beliefs, desires, and intentions. \Gls{DAI} and multi-agent systems extended these concepts to environments where multiple self-interested agents interact, introducing coordination through commitments and conventions, planning through shared and collaborative plans, and strategic interaction through utility functions, dominant strategies, and Nash equilibria. Software agents were presented as the practical translation of these theoretical foundations into autonomous programs operating within computational environments, classified into collaborative, interface, reactive, hybrid, and other types. Coding agents were then introduced as specialized \gls{LLM}-based software agents, consisting of planning, memory, perception, and action components, and capable of interpreting developer goals, generating code, and interacting with external tools. Context engineering was examined as the discipline of optimizing external context before presenting it to the model, encompassing techniques such as file ordering, chunking strategies, \gls{RAG}, and memory frameworks like Mem0. Tools were discussed beginning with self-supervised tool learning through Toolformer, progressing through fine-tuned tool-use models such as Gorilla and Tool\gls{LLM}, and culminating in the function calling paradigm and the \gls{MCP}, a universal open standard for defining, discovering, and invoking external tools that supports dynamic discovery, bi-directional communication, and access control. The \gls{CLI} was presented as a complementary tool interaction modality, tracing its origins from the Unix shell through the Bourne Shell and \gls{POSIX} standardization to \gls{BASH}, and reviewing research on natural language to \gls{BASH} translation and the design properties that account for the convergence of agentic coding tools on terminal-based interfaces. Skills were examined from their origins in robotics, where they were formalized as learned abilities to perform competent actions toward a goal, through their modern implementation as reusable behavioral abstractions encoded in natural language prompts and agentic coding manifests.

The related work section examined three areas closely aligned with this research. The first addressed \gls{ML}, reviewing both agents and platforms. \Gls{ML} agents were examined as a class of \gls{LLM}-based systems that automate \gls{ML} workflows, varying in pipeline coverage, platform integration, and tool interface design. Existing approaches either cover the complete \gls{AutoML} pipeline without integration into a \gls{ML} platform, employ tools but are limited to a single interface type or pipeline stage, or integrate within platform architectures without standardized tool interfaces. The landscape of \gls{ML} platforms was also surveyed, encompassing leaders such as Databricks, Google Cloud, Microsoft Azure, and \gls{AWS}, as well as challengers, visionaries, and niche players. Across the platforms reviewed, 96.15\% offer a \gls{CLI}, 76.92\% provide \gls{MCP} connectivity, 23.03\% enable hosting \gls{MCP} server, 76.92\% support agent hosting, 38.46\% provide skills, and 31\% include agents that assist in operating the platform. Despite increasing adoption, the effectiveness of \gls{ML} agents in industry remains underexplored. Benchmarking was reviewed across four areas. Coding agent benchmarks span safety, data science, repository-level code generation, automated program repair, communication skills, and collaborative coding, but none evaluates agents performing \gls{ML} tasks on \gls{ML} platforms. \Gls{MCP} benchmarks assess general-purpose tool interaction, domain-specific applications, and security, but do not target \gls{ML} pipeline management or compare against alternative tool types. \Gls{CLI} benchmarks remain limited to Terminal-Bench 2.0, which includes several \gls{ML}-related tasks but does not comprehensively cover the \gls{AutoML} pipeline or evaluate agents operating \gls{ML} platforms. \Gls{ML} agent benchmarks evaluate experimentation, engineering, and code generation tasks but do not assess agents managing pipelines on \gls{ML} platforms through tool interfaces. Tool design research focuses on optimizing tool selection and context management at scale but does not address the design of tools, skills, or agentic coding manifests for coding agents operating \gls{ML} platforms.

Across these three areas, several gaps emerge. No study has examined how \gls{CLI} tools should be constructed to be used effectively by coding agents operating \gls{ML} platforms. No benchmark evaluates coding agents managing feature, training, and inference pipelines on a \gls{ML} platform. It is also unclear how \gls{CLI} tools compare against \gls{MCP} tools when solving the same ML tasks, and no guidance exists on the conditions under which each tool type is more suitable for coding agents operating \gls{ML} platforms. While there is empirical evidence that skills improve coding agent performance, it remains unclear whether this effect holds equally across tool types, and in the context of \gls{ML}. This thesis addresses these gaps by answering the following three \glspl{RQ}:

\glsresetall
\begin{itemize}
\item What are the design requirements for \glspl{CLI} invoked by coding agents operating \gls{ML} platforms, and how do they relate to the \gls{MCP} as a tool calling protocol?
\item Which tool type, \gls{MCP} or \gls{CLI}, achieves a better trade-off between task success and token efficiency when coding agents manage feature, training, and inference pipelines on \gls{ML} platforms?
\item How do skills influence the performance of coding agents orchestrating \gls{ML} platforms across different types of \gls{ML} tasks, and does this influence differ between \gls{CLI} and \gls{MCP} tools?
\end{itemize}

\chapter{Method}

The goal of the experiments conducted in this thesis is to answer the three \glspl{RQ} 
defined in the previous chapter: the design requirements for \glspl{CLI} used by coding agents and their relation to \gls{MCP}, the comparison between \gls{CLI} and \gls{MCP} in terms of task success and token efficiency in \gls{MLOps}, and the effect of skills on agent performance across different tasks and interface types. These \glspl{RQ} are empirical in nature because the arguments raised so far are based on practitioner opinion rather than controlled evidence.

To investigate the first \gls{RQ}, an exploratory search of practitioner discussions 
is conducted to derive design patterns for \glspl{CLI} used by coding agents and to 
identify when each tool type is most appropriate. For the second \gls{RQ}, these 
patterns are combined with the Hopsworks Python \gls{SDK} to implement a \gls{CLI} 
and a \gls{MCP} server, which are then used by OpenCode to execute tasks drawn from 
\gls{MLE}-bench. Token consumption, completion time, task success, and the number of 
turns in which the agent invokes Hopsworks rather than executing local Python code 
are recorded and compared across both interfaces. For the third \gls{RQ}, the 
results are analyzed to identify turns in which the agent bypassed Hopsworks in 
favor of local Python execution. Skills are then derived from the Hopsworks 
documentation to address these gaps, and the same tasks are rerun to measure 
whether skills shift the agent toward greater Hopsworks use and how this affects 
performance across interface types and task categories.

This chapter begins by introducing experiments in \gls{SWE} as a research method and 
defining the terminology and process used throughout this study. It then describes

\glspl{RQ}.

\section{Experiments in Software Engineering}

\glsresetall

An experiment in \gls{SWE} is defined as ``an empirical enquiry that manipulates one factor or variable of the studied setting. Based on randomization, different treatments are applied to or by different subjects, while keeping other variables constant, and measuring the effects on the outcome variables. In human-oriented experiments, humans apply different treatments to objects, while in technology-oriented experiments, different technical treatments are applied to objects''~\cite{wohlin_experimentation_2024}. In essence, experiments are used to evaluate beliefs such as a theory or hypothesis by testing the relationship between a treatment and an outcome, thereby allowing researchers to draw conclusions about cause and effect \cite{bacon_novum_1995}.

Central to any experiment is the distinction between two types of variables~\cite{wohlin_experimentation_2024}. Independent variables, also known as factors, are the variables in the process that are controlled and manipulated by the experimenter. Dependent variables, also known as response variables, are those whose values are observed and measured to study the effect of changes in the independent variables. An experiment studies the effect of changing one or more independent variables while all other independent variables are held at a fixed level. A treatment is one particular value of a factor, and the choice of which treatments to apply constitutes the experiment design. Treatments are applied to combinations of objects, i.e. the entities being studied, and subjects, i.e. the participants or tools applying or receiving the treatments, if any. An experiment consists of a set of tests, where each test is a unique combination of a treatment, a subject, and an object. The number of tests carried out directly affects the experimental error, which is the estimate of how precisely one can measure the mean effect of any experimental factor. The experimental error, in turn, determines the confidence that can be placed in the results.

The experiment process consists of five main phases~\cite{wohlin_experimentation_2024}. The first phase is scoping, in which the goals and objectives of the experiment are established. The goal is formulated from the research problem, and a commonly suggested framework defines the scope in terms of the object of study, the purpose, the quality focus, the perspective from which the experiment is conducted, and the context in which it takes place.

The second phase is planning, in which the context is determined in detail, including the personnel involved and the environment. During planning, hypotheses are stated formally: the null hypothesis, which represents the default assumption of no effect, and the alternative hypothesis, which represents the expected effect. The independent and dependent variables are identified, and the experiment design is specified, including the instrumentation to be used. A critical aspect of planning is addressing validity, which has four dimensions. Internal validity concerns the reliability of the results and whether the observed effects can genuinely be attributed to the treatment rather than to confounding factors. External validity addresses how generalizable the results are beyond the specific experimental setting. Construct validity examines whether the treatment truly reflects the cause construct and whether the outcome adequately captures the effect construct. Conclusion validity concerns whether the observed relationship between treatment and outcome is statistically justified.

The third phase is operation, which encompasses three activities: preparation, execution, and data validation. Preparation involves readying the subjects and materials for the experiment. Execution is the phase in which the experiment is conducted according to the plan and design. Data validation involves checking the collected data for correctness and completeness before analysis.

The fourth phase is analysis and interpretation. In this phase, descriptive statistics are used to summarize the data, and the researcher considers whether the data set should be reduced, for example by removing outliers. A formal hypothesis test is then performed, and the results are interpreted. If the evidence is sufficient, the null hypothesis may be rejected in favor of the alternative hypothesis.

The fifth and final phase is presentation and packaging, in which the results are documented and compiled into a complete laboratory package that allows others to understand, evaluate, and replicate the experiment.

The remainder of this chapter applies the first four phases of this framework to 
the experiments conducted in this study. The fifth phase, presentation and packaging, 
is realized through this thesis itself.

\section{Scoping the Experiments}

The overall goal of the experiments is to empirically compare \gls{CLI}-based and 
\gls{MCP}-based tool interfaces for coding agents operating Hopsworks, and to study 
how the use of skills affects performance under each interface type. The object of 
study is coding agents operating Hopsworks, specifically when executing feature 
engineering, training, and inference tasks. The treatments correspond to the two 
interface types under investigation, and the presence or absence of skills 
constitutes a secondary treatment factor used in the second phase of the study. The 
purpose is comparative, aiming to measure differences in task success, token 
consumption, and completion time between the two interface types, as well as to 
evaluate the effect of introducing skills.

With the goal, object, treatments, and purpose established, the next phase 
specifies how the experiments are structured to deliver on this scope.


\section{Planning the Experiments}

The planning phase defines how the experiments are structured to answer the three 
\glspl{RQ} in a controlled and repeatable manner. It specifies the experimental 
setup, variables, hypotheses, procedures, and instrumentation, ensuring that 
observed differences can be attributed to the manipulated factors rather than 
external variation.

The first element of the plan is the context within which the experiments are 
conducted, including the infrastructure on which the testing system runs and the 
subjects and objects to which treatments are applied.

\subsection{Context and Setup}

The experiments are conducted within a controlled software-based testing system 
designed to evaluate coding agents operating in an \gls{MLOps} environment. The 
system defines the environment under evaluation, the subjects, and the objects of 
study, and provides the infrastructure required to execute repeated runs under 
consistent conditions. Figure~\ref{fig:mlops-agent} shows the architecture and the 
components that make up the system.

\begin{figure}[H]
    \centering

    \caption{Architecture of the testing system}

All components are deployed on Snurren, a \gls{GPU} development and testing cluster 
used by Hopsworks for internal experimentation and system development. The 
infrastructure stack is organized into four layers. At the hardware layer, the 
cluster is equipped with four NVIDIA \gls{RTX} A6000 \glspl{GPU}. A \gls{GPU} is a 
specialized processor that performs many parallel operations simultaneously and is 
widely used to accelerate the computations that dominate \gls{ML} training and 
inference workloads~\cite{harris_array_2020}. NVIDIA \gls{RTX} refers to a family of \glspl{GPU} that 
combine general-purpose parallel computation with hardware support for \gls{AI} and 
graphics workloads~\cite{nvidia_corporation_nvidia_2026}. At the \gls{OS} layer, the cluster runs Ubuntu, a 
Linux-based operating system used to manage hardware and system resources~\cite{canonical_ltd_enterprise_2026}. Linux is 
an open-source operating system kernel that manages hardware resources and provides 
the foundation on which user-facing software runs~\cite{torvalds_torvaldslinux_2026}. At the orchestration 
layer, Kubernetes runs on top of Ubuntu and serves as the container orchestration 
platform used to deploy and manage distributed applications across the 
cluster~\cite{the_kubernetes_authors_kuberneteskubernetes_2026}. At the application layer, the testing system is deployed as a 
Kubernetes pod named \texttt{mlops-agent}, located in a dedicated namespace named 
\texttt{mlops-agent-\gls{NS}}. A pod is the smallest deployable unit in Kubernetes 
and consists of one or more containers that share storage, network, and a 
specification for how to run~\cite{the_kubernetes_authors_pods_2026}. A \gls{NS} is a logical partition within a 
Kubernetes cluster used to scope and isolate resources such as pods and 
services~\cite{the_kubernetes_authors_namespaces_2026}.

At the center of the testing system is OpenCode, a coding agent that 
executes tasks by interacting with tools and external systems~\cite{anomaly_innovations_inc_anomalycoopencode_2026}. It receives 
tasks from \gls{MLE}-bench and executes them by interacting with Hopsworks, which 
runs within the same cluster and provides the \gls{MLOps} environment used for 
feature engineering, training, and inference. These interactions are performed 
through one of two interfaces, hops-\gls{CLI} or hops-\gls{MCP}. Skills are stored 
as a separate artifact called hops-skills within the agent environment, and are 
loaded into OpenCode to guide its behavior during execution. The coordinator manages the execution of experiments by assigning tasks, controlling run 
configurations, and tracking execution progress across the system.

Model access is handled through Lite\Gls{LLM}, a routing and abstraction layer that 
provides a unified interface for interacting with multiple \glspl{LLM}~\cite{litellm_berriailitellm_2026}. All 
model requests from OpenCode are routed through Lite\gls{LLM} to the configured 
\gls{LLM}. Lite\gls{LLM} also forwards all requests and responses to Langfuse, a 
logging and observability system used to track token usage, metadata, and execution 
traces for \gls{LLM} calls~\cite{langfuse_gmbh_langfuselangfuse_2026}. Hugging Face, an online platform that hosts 
open-source machine learning models, datasets, and tools~\cite{hugging_face_inc_hugging_2026}, provides the model 
weights for Kimi K2.6, which are loaded into the Hopsworks deployment, while the 
Claude Platform hosts Claude Opus 4.7.

The experiments are conducted entirely in software and require no human participants 
to apply treatments. The subjects are two \glspl{LLM}, Kimi K2.6 and Claude Opus 4.7, 
selected as recent \gls{SOTA} models from the open-source and closed-source segments 
of the \gls{LLM} landscape~\cite{openrouter_inc_kimi_2026}. Kimi K2.6 is open-source and runs within 
Hopsworks, while Claude Opus 4.7 is closed-source and is accessed via the 
Claude Platform~\cite{kimi_team_kimi_2026, anthropic_pbc_introducing_2026}. The objects are eight Kaggle competitions sourced from 
\gls{MLE}-bench~\cite{chan_mle-bench_2025}, listed in Table~\ref{tab:mle-bench-tasks}, covering seven \gls{ML} 
task categories. Image classification refers to assigning a discrete category to an 
input image. Image-to-image translation refers to mapping an input image to an 
output image, such as denoising or style transfer. Text classification refers to 
assigning a discrete category to an input text. Image regression refers to 
predicting a continuous-valued target from an input image. Audio classification 
refers to assigning a discrete category to an input audio signal. Tabular prediction 
refers to predicting a target value from structured tabular features. 
Sequence-to-sequence generation refers to mapping an input sequence to an output 
sequence of potentially different length, such as machine translation or text 
summarization.

With the testing system in place, the next planning element is the construction of 
the two interfaces under comparison.

\subsubsection{Interface Design Patterns}

A structured keyword search is conducted using Google Search and Hacker News. Hacker 
News is a social news website operated by Y Combinator on which users submit and 
discuss links related to computer science, software engineering, and technology, and 
which serves as a primary venue for practitioner discussion in these areas~\cite{y_combinator_management_llc_hacker_2026}. 
Y Combinator is a startup accelerator that provides early-stage funding and 
mentorship to technology companies and operates Hacker News as part of its broader 
engagement with the technology community~\cite{y_combinator_management_llc_y_2026}. The queries are 
\texttt{"MCP AND CLI"}, \texttt{"model context protocol" AND MLOps}, and 
\texttt{CLI AND (agent OR agentic)}. The first query targets practitioner 
discussion on \gls{CLI} and \gls{MCP}. The second focuses on \gls{MCP} usage in 
\gls{MLOps}. The third targets \gls{CLI} design for agentic systems. 
The first five result pages are reviewed per query for both Google Search and Hacker 
News, due to time constraints and diminishing returns in relevance beyond this range. 
For Hacker News, all discussions returned within these five pages are inspected, 
including linked external sources referenced within those discussions. Ranking order 
is not used in selection.

Relevant sources are screened, non-English results are filtered, and duplicates are 
removed. Design patterns are extracted through qualitative coding of the selected 
material. The coding process is used to identify recurring structures in agent 
interaction design for both \gls{CLI} and \gls{MCP} systems, as well as criteria for 
distinguishing when each interface type is appropriate.

These patterns inform the prompts that drive the generation of the two interfaces.

\subsubsection{Prompt Engineering}

The extracted patterns are used to construct two prompts: one for generating the 
\gls{CLI} interface and one for generating the \gls{MCP} server. The \gls{CLI} 
prompt incorporates \gls{CLI}-specific interaction patterns, while the \gls{MCP} 
prompt incorporates the Scale\gls{MCP} framework~\cite{marcelloni_scalemcp_2026}. 
Both prompts share the same Hopsworks source material and differ only in 
interface-specific design constraints.

With the two interfaces in place, the experiments can compare them under controlled 
conditions. The variables that are manipulated and measured during this comparison 
are defined next.

\subsection{Variables}

The independent variables are interface type, with two levels \gls{CLI} and 
\gls{MCP}, and skill usage, with two levels without skills and with skills examined 
in the second phase of the experiment. The \gls{LLM} backend is treated as an 
additional factor with two levels Kimi K2.6 and Claude Opus 4.7. The \gls{ML} task 
category is treated as a blocking variable.

The dependent variables are task success measured as the \gls{MLE}-bench raw 
competition score and medal classification, token consumption per run, task 
completion time as wall-clock time per run, and platform usage measured as the 
number of agent turns in which the agent invokes a Hopsworks platform operation 
through the assigned interface rather than executing local Python code. All other 
factors, including prompts, agent configuration, decoding parameters, and execution 
environment, are held constant to isolate the effects of the independent variables.

These variables enable the formal statement of the hypotheses that the experiments 
test.

\subsection{Hypotheses}

For the interface comparison, the null hypothesis states that there is no difference 
between \gls{CLI} and \gls{MCP} in terms of task success, token consumption, or 
completion time. The alternative hypothesis states that at least one of these 
metrics differs between the two interface types. For the skills evaluation, the null 
hypothesis states that adding skills does not change agent performance under either 
interface. The alternative hypothesis states that skills change performance and that 
the effect may vary across interface types and task categories.

Testing these hypotheses requires a design that arranges the variables into a 
concrete set of runs.

\subsection{Experiment Design}

The experiments are organized into two sequential phases. The first phase compares 
the two interface types without skills, varying interface type, \gls{LLM} backend, 
and task category, and replicating each combination three times. This yields 
$2 \times 2 \times 8 \times 3 = 96$ tests in the first phase.

Between the two phases, a static skill set is produced through a two-step process. 
An initial skill set is first generated from the Hopsworks documentation, encoding 
the platform operations and recommended usage patterns that the agent is expected 
to apply when solving \gls{ML} tasks. The runs from the first phase are then 
analyzed to identify turns in which the agent executed local Python code rather 
than invoking the corresponding Hopsworks operation through the assigned interface. 
Each such instance is matched to the appropriate Hopsworks endpoint, and the 
initial skills are extended where coverage is missing.

The second phase repeats the same set of treatment combinations with the resulting 
skill set enabled, yielding a further $96$ tests. The skill condition is therefore 
a between-phase factor rather than a within-phase factor, since the skill set used 
in the second phase is itself derived from analysis of the first phase. In total, 
the design comprises $192$ tests. The three repetitions per cell allow within-cell 
variance to be estimated and provide a basis for assessing the stability of results 
in the presence of \gls{LLM} non-determinism.

Each of these tests follows a fixed protocol that ensures runs are executed under 
the same conditions.

\subsection{Execution Protocol}

Each run is executed according to a fixed protocol. The \texttt{mlops-agent} pod is reset before each run so that no artifacts persist from 
previous runs and OpenCode starts each run from a clean state. Each run is assigned 
its own dedicated Hopsworks project, which isolates the data, features, models, and 
other platform artifacts produced during the run from those of any other run. The 
\gls{MLE}-bench task description, dataset, and validation tooling are made 
available in the workspace. The agent is instructed to produce a 
\texttt{submission.csv} file for the assigned competition and is given access to 
Hopsworks exclusively through the assigned interface. No other external tools are 
provided. A timestamp is recorded at the start of the run. Each run has a 
wall-clock time limit of two hours, which is the only stopping condition imposed. 
Within this limit the agent operates autonomously and is free to retry, recover 
from failures, and revise its approach as many times as it sees fit. A run 
terminates when the agent finishes on its own or when the two-hour limit is 
reached. A final timestamp and total token count are recorded at termination.

Run order across the full set of treatment combinations is randomized using a fixed 
random seed to reduce ordering effects while preserving reproducibility of the run 
schedule. The randomization is applied once across the entire experiment rather 
than re-applied per interface or model.

Beyond the execution protocol, additional measures are taken to limit variation 
introduced by the underlying \gls{LLM}.

\subsection{Stochasticity Controls}

Because the agent is driven by an \gls{LLM}, run-to-run variation cannot be 
eliminated but is constrained as far as practical. The decoding temperature is 
fixed at zero across all runs to minimize sampling-induced variation. The model 
versions are pinned: Kimi K2.6 is served from a fixed snapshot deployed within 
Hopsworks, and Claude Opus 4.7 is accessed via an explicit version-locked model 
identifier on the Claude Platform. The system prompt, agent configuration, and tool 
descriptions are identical across runs for a given condition. The remaining 
variation attributable to \gls{LLM} non-determinism is captured by the three 
repetitions per cell.

The data needed to evaluate each run is captured by the instrumentation built into 
the testing system.

\subsection{Instrumentation}

All runs are logged automatically by a combination of OpenCode, the coordinator 
service, \gls{MLE}-bench, Lite\gls{LLM}, and Langfuse, with the coordinator service 
acting as the central aggregation point. OpenCode records the full sequence of 
agent actions, tool calls, and tool responses for each run. \gls{MLE}-bench records 
the validation status of each submission and the resulting raw score and medal 
classification. Lite\gls{LLM} provides per-call token usage statistics for all 
\gls{LLM} requests routed through it, and Langfuse provides cost calculations 
derived from the token usage and the corresponding model pricing. The coordinator 
service records the assigned interface, model, task, and skill condition for each 
run, tracks start and end timestamps, monitors run status, and queries Lite\gls{LLM} 
and Langfuse to retrieve token usage and cost information at the end of each run. 
All collected data is consolidated by the coordinator and persisted in the 
Hopsworks feature store, where the records are organized as feature groups for 
subsequent analysis. Logs from all sources are linked through a unique run 
identifier encoding the interface, model, task, skill condition, and repetition 
index. Analysis is conducted within Jupyter notebooks running against Hopsworks, 
which natively supports notebook-based exploration of data stored in the feature 
store.

The remaining planning element is the assessment of validity, which considers the 
extent to which the design supports sound conclusions.

\subsection{Validity}

Internal validity is threatened by the possibility that differences in submission 
score, token consumption, or elapsed time reflect prompt sensitivity or stochastic 
variation in generation rather than a genuine effect of interface type. This is 
partially mitigated by using the same base prompt structure and decoding parameters 
across conditions and by treating \gls{LLM} backend as a factor so that 
model-specific effects can be separated from interface effects. Remaining variance 
attributable to \gls{LLM} non-determinism constitutes irreducible experimental error.

External validity is limited by the scope of the benchmark. The eight Kaggle 
competitions represent a curated subset of competitive \gls{ML} tasks and may not 
generalize to production \gls{MLOps} workflows, proprietary datasets, or agent 
frameworks other than OpenCode. Generalization to \gls{MLOps} platforms other than 
Hopsworks is also not directly supported by these experiments.

Construct validity depends on whether \gls{MLE}-bench submission score is an 
adequate operationalization of \gls{ML} solution quality. Because \gls{MLE}-bench 
scores are derived from the same evaluation protocol as the original Kaggle 
competitions, they are considered a reasonable proxy for competitive \gls{ML} 
performance. Token consumption is a direct count of tokens processed and requires 
no further operationalization. Elapsed time is recorded as wall-clock time and 
therefore reflects not only agent reasoning but also infrastructure latency, which 
is acknowledged as a limitation of this measure. Platform usage is measured by 
counting turns in which the agent executes local Python code rather than invoking 
a platform operation, and depends on the accuracy with which these turns can be 
classified from the agent logs.

Conclusion validity is strengthened by the inclusion of three repetitions per cell, 
which allows within-cell variance attributable to \gls{LLM} non-determinism to be 
estimated and separated from between-condition differences. With eight task 
categories and three repetitions, each cell contains 24 observations when 
aggregated across task categories, providing reasonable power to detect moderate 
effect sizes. Conclusions for individual task categories are based on three 
observations per cell and should be interpreted with greater caution.

With the experiments fully planned, the next phase concerns putting the plan into 
practice.
